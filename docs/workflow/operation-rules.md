# 그룹웨어 봇팀 운영 규칙

## 안전 원칙

- 민감정보를 출력, 저장, 커밋, 전송하지 않는다.
- `.env`, `auth.json`, OAuth 세션, 플랫폼 토큰, API key는 벤치마킹 대상이 아니다.
- root 권한, systemd, DB, 배포, 외부 공개 작업은 사용자 승인 범위 안에서만 진행한다.

## 요청 처리 원칙: 아리아 경유

1. 아리아가 그룹웨어 계정사용자 요청을 받는다.
2. 아리아가 대장에게 요약 보고한다.
3. 대장이 아리아 단계 전달을 승인하면 아리아가 싱드에게 전달한다.
4. 싱드는 전달받은 내용을 대장에게 다시 보고하고 오케스트레이션 승인 요청을 한다.
5. 대장이 싱드 단계 실행을 승인하면 싱드는 Kanban 작업 그래프를 만든다.
6. 역할별 봇이 작업한다.
7. 싱드가 최종 취합 보고한다.

## 요청 처리 원칙: 대장 → 싱드 직접 요청

1. 대장이 싱드에게 개발/운영 요청을 직접 전달한다.
2. 싱드는 바로 실행하지 않고 요청 요약, 작업 범위, 제외 범위, 위험/승인 필요 사항, 예상 파이프라인을 대장에게 보고한다.
3. 싱드는 대장에게 오케스트레이션 실행 승인을 요청한다.
4. 대장이 승인하면 싱드는 Kanban 작업 그래프를 만든다.
5. 역할별 봇이 작업한다.
6. 싱드가 최종 취합 보고한다.

아리아 경유든 대장 직접 요청이든, 싱드의 실제 실행 기준은 동일하다. **대장 실행 승인 전에는 preview/status/검토만 가능하고, Kanban 카드 생성·dispatch·구현 착수는 하지 않는다.**

## 검증 원칙

- 완료 보고 전 실제 명령 결과로 확인한다.
- `running` 표시만 보고 실제 작업 중이라고 단정하지 않는다.
- Kanban 작업은 `show`, `runs`, `log`, `list`, `dispatch --dry-run`으로 확인한다.

## GitHub 자동화 원칙

- 승인된 오케스트레이션 범위 안에서는 branch 생성, commit, PR 생성, CI 확인, PR merge, 원격/로컬 branch 삭제까지 GitHub release gate에 포함한다.
- merge 전에는 최신 head 기준 CI/check가 통과했는지 확인한다.
- merge/delete 후에는 기본 브랜치 상태와 남은 브랜치를 확인한다.
- 배포, 유료 리소스, 외부 공개, 비밀값 입력/교체, 운영 DB 실데이터 변경은 GitHub 자동화 범위에 포함하지 않고 별도 승인으로 둔다.
