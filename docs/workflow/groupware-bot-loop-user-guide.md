# 그룹웨어 봇 루프 사용 안내

## 한 줄 요약

그룹웨어 봇 루프는 사용자가 싱드에게 일을 말하면, 싱드가 뒤에서 역할별 내부 봇에게 일을 나누고 확인까지 거친 뒤 다시 쉬운 말로 보고하는 작업 방식입니다.

## 사용자가 기억할 핵심 3가지

1. 평소에는 싱드하고만 대화하면 됩니다.
2. 싱드는 필요한 내부 봇을 알아서 불러 계획, 실행, 검토, 테스트, 문서화를 나눠 맡깁니다.
3. 배포, 삭제, 외부 전송, 비용이 드는 작업, 비밀값 입력 같은 위험한 일은 사용자 확인 전에는 진행하지 않습니다.

## 왜 처음에는 작은 문서 작업부터 테스트하나요?

지금 그룹웨어 봇 루프는 기본 자동화 흐름이 준비된 상태입니다. 다만 처음부터 큰 기능 개발이나 배포를 돌리기보다, 위험이 낮은 작은 문서 작업으로 처음부터 끝까지 잘 이어지는지 먼저 확인하는 단계입니다. 긴 작업이나 큰 변경은 중간 확인이 필요할 수 있습니다.

쉽게 말하면, 큰 공사 전에 작은 일감으로 팀 호흡을 먼저 맞춰보는 테스트입니다.

## 작업이 돌아가는 흐름

보통은 아래 순서로 돌아갑니다.

1. 계획: 무엇을 어디까지 할지 정합니다.
2. 실행: 필요한 경우 실제 파일이나 코드를 고칩니다.
3. 리뷰: 빠진 점이나 위험한 점이 없는지 확인합니다.
4. 테스트: 정말 되는지 확인합니다.
5. 문서화: 나중에 다시 볼 수 있게 정리합니다.
6. 최종 보고: 싱드가 한 일, 근거, 남은 위험, 다음 액션을 쉬운 말로 알려줍니다.

사용자는 이 과정을 모두 직접 관리할 필요가 없습니다. 싱드가 중간 정리를 맡고, 필요한 경우에만 추가 질문을 합니다.

## 싱드에게 어떻게 요청하면 되나요?

명령어를 외울 필요는 없습니다. 평소에는 자연어로 말하면 됩니다.

예시:

- "싱드, 업무 요청 기능 MVP 범위 정리해줘."
- "싱드, 지금 문서랑 실제 구현이 맞는지 확인해줘."
- "싱드, 작은 문서 작업으로 봇 루프 테스트 한 번 돌려줘."
- "싱드, 배포 준비는 해도 되는데 실제 공개 전에는 꼭 나한테 확인받아."

싱드가 할 일:

- 요청의 목표를 정리합니다.
- 범위 밖 작업이 없는지 봅니다.
- 필요한 내부 봇에게 작업을 나눕니다.
- 결과를 모아 다시 설명합니다.

## 내부 봇은 어떤 일을 하나요?

사용자가 이름을 외울 필요는 없지만, 뒤에서는 이런 역할로 나뉩니다.

- 기획: 목표와 범위를 정리합니다.
- 실행: 실제 수정 작업을 합니다.
- 리뷰: 빠진 점과 위험 요소를 확인합니다.
- 테스트: 정상 동작, 실패, 미확인 여부를 확인합니다.
- 문서화: 결과를 사람이 읽기 쉬운 문서로 정리합니다.

중요한 점은, 내부 봇이 바로 사용자에게 말하는 것이 아니라 싱드가 결과를 모아 최종 보고한다는 것입니다.

## 안전하게 멈추는 경우

아래처럼 위험하거나 되돌리기 어려운 일은 바로 진행하지 않고 멈춘 뒤 사용자 확인을 받습니다.

- 실제 배포
- 파일 삭제 또는 대규모 이동
- 외부 서비스로 데이터 전송
- 비용이 발생할 수 있는 작업
- 비밀키나 인증정보 입력
- 공개 전환처럼 되돌리기 어려운 운영 변경

또한 결과가 불확실하면 억지로 완료로 처리하지 않습니다.

- 실패한 것은 실패라고 말합니다.
- 아직 확인 못 한 것은 미확인이라고 말합니다.
- 추가 판단이 필요하면 싱드가 선택지를 정리해 다시 묻습니다.

## 첫 실전 테스트에서 확인할 것

이번 첫 테스트는 작은 문서 작업으로 그룹웨어 봇 루프가 실제로 한 바퀴 도는지 확인하는 용도입니다.

성공 기준은 아래와 같습니다.

- Kanban 작업 카드가 만들어집니다.
- 담당 봇이 작업을 잡습니다.
- 문서 결과가 남습니다.
- 리뷰 또는 확인 결과가 남습니다.
- 싱드가 최종 보고에 쓸 정보를 모읍니다.
- 실패하거나 미확인인 경우 숨기지 않고 다음 조치를 남깁니다.

즉, 목표는 "완벽한 대형 작업"이 아니라 "작은 작업이 끊기지 않고 끝까지 이어지는지 확인"하는 것입니다.

## 지금 기준으로 이해하면 안전한 표현

현재 문서들을 종합하면, 그룹웨어 봇 루프는 "자동화 기반은 준비되었고, 작은 작업으로 첫 완주를 확인하는 단계"라고 이해하면 가장 안전합니다.

즉,

- 아직 아무 검증 없이 큰 위험 작업을 자동으로 맡기는 단계는 아닙니다.
- 그렇다고 수동 테스트만 있는 상태도 아닙니다.
- 작은 실제 작업으로 계획부터 최종 보고까지 연결되는지 확인하는 중입니다.

## 싱드가 최종 보고할 때 보통 포함하는 내용

싱드의 최종 보고는 보통 아래 순서로 읽으면 됩니다.

- 한 일
- 확인한 근거
- 남은 위험 또는 미확인 사항
- 다음에 하면 좋은 일

사용자는 이 네 가지만 봐도 지금 어디까지 끝났는지 빠르게 파악할 수 있습니다.
