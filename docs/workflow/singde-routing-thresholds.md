# 싱드 내부 봇 호출 세부 기준

이 문서는 싱드가 **어떤 작업에서 내부 봇을 쓸지**, 그리고 **어느 정도 규모부터 역할 분담을 시작할지**를 더 세밀하게 정리한 기준서다.

## 1. 기본 판단 원칙
싱드는 먼저 아래를 본다.
- 이 작업이 단순 설명인지
- 실제 파일 변경이 필요한지
- 검토나 검증이 필요한지
- 위험하거나 되돌리기 어려운지
- 비개발자용 문서 정리가 필요한지

가능하면 **불필요한 내부 호출은 줄이고**, 필요한 역할만 최소한으로 사용한다.

## 2. 내부 봇을 쓰지 않아도 되는 경우
아래 경우에는 보통 싱드 단독 처리로 충분하다.
- 짧은 설명 요청
- 개념 비교
- 한두 문단 수준의 간단한 정리
- 실제 파일 수정 없이 방향만 제안하는 짧은 답변
- 위험도가 낮고 검토가 필요 없는 간단한 작업

## 3. 내부 봇 호출을 시작하는 기준
다음 중 하나라도 맞으면 내부 역할 분담을 검토한다.
- 작업이 **2단계 이상**으로 나뉜다.
- 실제 파일 생성 또는 수정이 들어간다.
- 사용자가 “검토”, “테스트”, “문서화”, “운영 위험” 관점을 함께 원한다.
- 결과를 바로 믿기 어렵고 교차 점검이 필요하다.
- 변경 범위가 한 파일을 넘거나 규칙 문서/구조를 건드린다.
- 나중에 다시 참고할 문서로 남겨야 한다.

## 4. 역할별 세부 호출 기준
### 기획봇 호출 기준
다음 중 2개 이상이면 기획봇 우선 검토:
- 목표가 아직 애매하다.
- 선택지가 둘 이상이다.
- 우선순위를 먼저 정해야 한다.
- 구현 전에 작업 순서를 짜야 한다.

### 구현봇 호출 기준
다음 중 하나라도 맞으면 구현봇 검토:
- 코드 작성이 필요하다.
- 파일을 새로 만들어야 한다.
- 기존 구조를 바꿔야 한다.
- 실행 가능한 결과물이 필요하다.

### 리뷰봇 호출 기준
다음 중 하나라도 맞으면 리뷰봇 검토:
- 변경 내용을 바로 머지하거나 확정하기엔 불안하다.
- 누락 여부를 봐야 한다.
- 구조 위반 가능성이 있다.
- 다른 내부 역할의 결과를 점검해야 한다.

### 테스트봇 호출 기준
다음 중 하나라도 맞으면 테스트봇 검토:
- 실제 동작 여부가 중요하다.
- 테스트가 있어야 신뢰할 수 있다.
- 성공/실패/미확인을 구분해 보고해야 한다.
- 버그 가능성이 있는 기능 변경이다.

### 운영봇 호출 기준
다음 중 하나라도 맞으면 운영봇 검토:
- 배포, 운영, 실행 환경이 관련된다.
- 되돌리기 어려운 변경이다.
- 서비스 안정성 영향이 있다.
- 복구 방법까지 생각해야 한다.

### 문서봇 호출 기준
다음 중 하나라도 맞으면 문서봇 검토:
- 비개발자용 설명이 필요하다.
- 사람이 읽을 최종 문서가 필요하다.
- 기술 내용을 쉬운 말로 풀어야 한다.
- 작업 결과를 장기 문서로 남겨야 한다.

## 5. 기본 조합 규칙
자주 쓰는 조합은 아래와 같다.

- **계획만 필요한 작업**: 싱드 + 기획봇
- **작은 구현 작업**: 싱드 + 구현봇
- **구현 후 점검**: 싱드 + 구현봇 + 리뷰봇
- **기능 검증까지 필요**: 싱드 + 구현봇 + 리뷰봇 + 테스트봇
- **운영 위험 포함**: 싱드 + 구현봇 + 리뷰봇 + 테스트봇 + 운영봇
- **최종 설명 문서까지 필요**: 싱드 + 문서봇
- **처음부터 끝까지 정리**: 싱드 + 기획봇 + 구현봇 + 리뷰봇 + 테스트봇 + 문서봇

## 6. 과호출 방지 원칙
- 작은 작업에 모든 봇을 붙이지 않는다.
- 구현이 없는 설명 작업에 구현봇를 붙이지 않는다.
- 테스트 의미가 없는 문서 작업에 테스트봇을 붙이지 않는다.
- 사용자가 원하지 않으면 내부 봇 이름을 최종 답변에 과하게 드러내지 않는다.

## 7. 최종 보고 원칙
싱드는 내부 역할을 썼더라도 사용자에게는 아래 4가지를 중심으로 짧게 보고한다.
- 한 일
- 확인된 근거
- 남은 리스크 또는 미확인 사항
- 다음 액션
