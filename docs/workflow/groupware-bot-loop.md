# 그룹웨어 봇 루프 운영 문서

## 결론

그룹웨어 봇 루프는 **대장의 요청을 싱드가 작업으로 쪼개고, 역할별 봇이 실행하고, 검토·테스트에서 떨어지면 다시 수정시키고, 통과하면 문서화해서 싱드가 최종 보고하는 반복 공정**이다.

이 문서는 그룹웨어 봇 팀이 “한 번 답하고 끝나는 챗봇”이 아니라, **계획 → 실행 → 검토 → 수정 → 재검토 → 보고**를 계속 돌리는 개발 팀처럼 움직이게 하기 위한 운영 기준이다.

---

## 1. 기본 원칙

- 사용자는 원칙적으로 **싱드(`singde`)하고만 대화한다**.
- 싱드는 사용자의 요청을 정리하고, 필요한 내부 봇에게 작업을 나눈다.
- 내부 봇은 사용자에게 직접 최종 보고하지 않는다.
- 내부 봇 결과는 싱드가 다시 확인하고, 쉬운 한국어로 통합 보고한다.
- 검증되지 않은 결과는 완료로 보지 않는다.
- 카드 작업범위에 배포/릴리즈/merge/branch cleanup/release gate가 명시되어 있으면 해당 항목은 승인된 것으로 진행하되, 카드에 명시되지 않은 삭제, 외부 전송, 도메인/DNS, 유료 리소스, 비밀값 입력, production DB 실데이터 변경은 별도 승인 전에는 실행하지 않는다.

---

## 2. 루프의 전체 흐름

```text
대장 요청
↓
싱드가 목표와 조건 정리
↓
작업 유형 판단
↓
Kanban 카드 생성
↓
역할별 봇에게 배정
↓
각 봇이 작업 수행
↓
리뷰/테스트/문서 검증
↓
문제 있으면 수정 카드 생성
↓
통과하면 최종 보고 카드로 이동
↓
싱드가 대장에게 결과 보고
```

핵심은 **실패했을 때 멈추지 않고, 실패를 다음 수정 작업으로 바꾸는 것**이다.

---

## 3. 봇별 역할

### 싱드 `singde` — 총괄 루프

싱드는 중앙 조율자다.

담당:

- 사용자 요청 정리
- 작업 유형 판단
- 작업 카드 생성
- 내부 봇 결과 취합
- 충돌·미확인 사항 확인
- 대장에게 최종 보고

싱드는 필요한 경우에만 직접 처리한다. 실질적인 기획, 구현, 리뷰, 테스트, 문서화는 가능한 한 역할별 봇에게 맡긴다.

### 기획봇 `gwplanner` — 기획 루프

담당:

- 목표와 범위 정리
- 우선순위 판단
- 구현 전 계획 작성
- 요구사항의 빠진 부분 확인
- 기술 선택지가 있을 때 장단점 정리

대표 산출물:

- 작업 계획
- 요구사항 정리
- MVP 범위
- 구현 전 체크리스트

### 구현봇 `gwbuilder` — 구현 루프

담당:

- 코드 작성
- 파일 수정
- 구조 변경
- 기능 구현
- 오류 원인 분석과 수정

대표 루프:

```text
관련 파일 확인
↓
테스트 또는 재현 조건 확인
↓
최소 범위 구현
↓
실행/테스트
↓
실패 시 원인 분석
↓
수정 후 재검증
```

### 리뷰봇 `gwreviewer` — 리뷰 루프

담당:

- 요구사항 충족 여부 확인
- 누락된 조건 점검
- 코드 품질 점검
- 보안·위험 요소 점검
- 과도한 구현 또는 엉뚱한 변경 확인

리뷰가 실패하면 완료가 아니라 **수정 카드**로 이어진다.

### 테스트봇 `gwtester` — 테스트 루프

담당:

- 테스트 실행
- 실패 재현
- 정상/실패/미확인 구분
- 검증 명령과 결과 정리
- 배포 URL 또는 실행 화면 확인

테스트 결과는 반드시 아래처럼 나눠야 한다.

- 통과
- 실패
- 미확인
- 확인 불가 사유

### 문서봇 `gwdocs` — 문서 루프

담당:

- 작업 결과 문서화
- 비개발자용 설명 정리
- 변경 내용 요약
- 사용법/운영법 정리
- 문서와 실제 결과의 불일치 확인

### 운영봇 `gwops` — 운영·배포 루프

담당:

- 배포 준비 점검
- 실행 환경 점검
- 호스팅/도메인/환경변수 위험 확인
- 운영 안정성 확인
- 장애 복구 관점 점검

운영봇 작업은 외부 공개나 비용 발생 가능성이 있으므로, 위험 작업은 사용자 승인 게이트를 반드시 둔다.

## 4. 작업 유형별 기본 루프

## 4.1 기능 개발 루프

```text
기획 → 구현 → 리뷰 → 테스트 → 문서화 → 최종 보고
기획봇 → 구현봇 → 리뷰봇 → 테스트봇 → 문서봇 → 싱드
```

사용 예:

```bash
cd /home/wrhrgw/gw
./scripts/groupware-auto-workflow.sh \
  --type feature \
  "작업 요청 접수 기능 v1" \
  "사용자가 싱드에게 개발 요청을 말하면 목표, 필요한 것, 미정 사항, 다음 액션으로 정리하는 기능을 설계하고 구현한다."
```

완료 조건:

- 기획 산출물이 있다.
- 구현 변경이 명확하다.
- 리뷰가 통과했다.
- 테스트 결과가 확인됐다.
- 문서가 갱신됐다.
- 싱드가 대장에게 쉬운 말로 최종 보고했다.

## 4.2 버그 수정 루프

```text
재현/원인 정리 → 최소 수정 → 리뷰 → 재테스트 → 문서/보고
기획봇 또는 테스트봇 → 구현봇 → 리뷰봇 → 테스트봇 → 문서봇/싱드
```

사용 예:

```bash
cd /home/wrhrgw/gw
./scripts/groupware-auto-workflow.sh \
  --type bugfix \
  "로그인 실패 원인 수정" \
  "정상 계정인데 로그인 실패가 나는 문제를 재현하고 최소 범위로 수정한다."
```

완료 조건:

- 실패 재현 조건이 기록됐다.
- 원인 또는 가장 가능성 높은 원인이 정리됐다.
- 수정 범위가 최소화됐다.
- 같은 문제가 다시 발생하지 않는 검증이 있다.

## 4.3 문서 작업 루프

```text
문서 범위 정리 → 문서 작성 → 일관성 검토 → 쉬운 설명 정리 → 최종 보고
기획봇 → 문서봇 → 리뷰봇 → 문서봇 → 싱드
```

사용 예:

```bash
cd /home/wrhrgw/gw
./scripts/groupware-auto-workflow.sh \
  --type docs \
  "그룹웨어 봇 사용법 문서" \
  "비개발자도 이해할 수 있게 싱드 중심 작업 요청 방법을 정리한다."
```

완료 조건:

- 문서 위치가 명확하다.
- 실제 스크립트/설정과 문서가 맞다.
- 대장이 이해할 수 있는 표현으로 정리됐다.

## 4.4 리뷰/검증 루프

```text
검토 범위 정리 → 리뷰 → 테스트 확인 → 문제별 수정 카드 생성 또는 완료 보고
기획봇 → 리뷰봇 → 테스트봇 → 싱드
```

사용 예:

```bash
cd /home/wrhrgw/gw
./scripts/groupware-auto-workflow.sh \
  --type review \
  "최근 변경 검토" \
  "변경 범위가 요구사항을 만족하는지 리뷰하고 검증한다."
```

완료 조건:

- 검토 대상이 명확하다.
- 통과/실패/미확인이 구분됐다.
- 실패 항목은 수정 카드로 이어졌다.

## 4.5 배포 루프

```text
배포 기획 → 배포 준비 → 위험 리뷰 → 배포 범위 확인/승인 게이트 → 배포 실행/운영 점검 → 배포 검증 → 문서화 → 최종 보고
기획봇 → 구현봇 → 리뷰봇 → 싱드 → 운영봇 → 테스트봇 → 문서봇 → 싱드
```

사용 예:

```bash
cd /home/wrhrgw/gw
./scripts/gw-auto-workflow.sh \
  --type deploy \
  "그룹웨어 데모 웹앱 배포" \
  "현재 웹앱을 외부 URL로 확인 가능하게 준비한다. 카드 작업범위에 실제 배포가 명시되면 승인된 것으로 진행하고, DNS/유료/비밀값/production DB 작업은 별도 승인으로 분리한다."
```

배포 루프의 특별 규칙:

- 실제 배포/릴리즈가 카드 작업범위에 명시되어 있는지 먼저 확인한다.
- 비밀값은 출력하거나 문서에 저장하지 않는다.
- 카드에 명시되지 않은 유료 리소스, 도메인/DNS 연결, 외부 전송, production DB 실데이터 변경은 별도 승인 없이 진행하지 않는다.
- 배포 후 URL, 상태, 로그, 롤백 가능성을 확인한다.

## 5. 작업 카드 표준 형식

모든 Kanban 카드는 가능하면 아래 형식을 따른다.

```text
목표:
- 이 카드가 끝나면 무엇이 달라지는가?

담당 봇:
- singde / gwplanner / gwbuilder / gwreviewer / gwtester / gwdocs / gwops

입력 문서/참고 파일:
- 관련 문서, 코드, 스크립트 경로

작업 범위:
- 해야 할 일

하지 말아야 할 것:
- 범위 밖 작업
- 위험 작업
- 카드 작업범위에 명시되지 않아 별도 승인 전 금지되는 작업

완료 조건:
- 완료라고 판단할 수 있는 구체 기준

검증 방법:
- 실행할 명령
- 확인할 파일
- 기대 결과

결과 보고 형식:
- 한 일
- 확인한 근거
- 실패/미확인 사항
- 다음 액션
```

## 6. 단계별 게이트

게이트는 “다음 단계로 넘어가기 전 반드시 확인해야 하는 문”이다.

### 6.1 기획 게이트

통과 조건:

- 목표가 한 문장으로 정리됐다.
- MVP 범위와 제외 범위가 구분됐다.
- 미정 사항이 숨겨지지 않았다.
- 다음 구현자가 추측하지 않아도 된다.

실패 시:

- 기획봇에게 보완 카드 생성
- 필요하면 싱드가 대장에게 질문

### 6.2 구현 게이트

통과 조건:

- 요구사항에 맞는 파일이 수정됐다.
- 변경 범위가 과도하지 않다.
- 실행 또는 테스트 근거가 있다.
- 실패한 명령이 있으면 숨기지 않았다.

실패 시:

- 구현봇에게 수정 카드 생성
- 원인 불명확 시 테스트봇에게 재현 카드 생성

### 6.3 리뷰 게이트

통과 조건:

- 명세 충족 여부가 확인됐다.
- 품질/보안/권한 문제가 점검됐다.
- 문제 항목은 구체적으로 적혔다.

실패 시:

- 수정 카드 생성
- 수정 후 리뷰봇이 재리뷰

### 6.4 테스트 게이트

통과 조건:

- 테스트 또는 실행 확인이 끝났다.
- 통과/실패/미확인이 분리됐다.
- 미확인 사유가 적혔다.

실패 시:

- 실패 로그와 재현 조건을 남김
- 구현봇에게 수정 카드 생성

### 6.5 문서 게이트

통과 조건:

- 결과 문서가 실제 변경과 맞다.
- 대장이 이해할 수 있는 한국어로 정리됐다.
- 다음 액션이 명확하다.

실패 시:

- 문서봇에게 문서 보완 카드 생성

### 6.6 사용자 승인 게이트

필요한 경우:

- 외부 배포
- 파일 삭제 또는 대규모 이동
- 비밀값 입력
- 유료 리소스 사용
- 외부 서비스 전송
- 되돌리기 어려운 운영 변경

통과 조건:

- 싱드가 대장에게 위험과 선택지를 설명했다.
- 대장이 명시적으로 승인했다.

실패 또는 보류 시:

- 카드를 `blocked` 또는 `scheduled` 상태로 둔다.
- 후속 위험 작업은 시작하지 않는다.

## 7. 실패 시 재작업 규칙

실패는 종료가 아니라 다음 루프의 입력이다.

```text
테스트 실패
→ 실패 로그와 재현 조건 정리
→ 구현봇 수정 카드 생성
→ 테스트봇 재테스트

리뷰 실패
→ 리뷰봇이 문제 항목 정리
→ 구현봇 수정 카드 생성
→ 리뷰봇 재리뷰

문서 불일치
→ 문서봇 보완 카드 생성
→ 리뷰봇 또는 싱드 확인

요구사항 불명확
→ 기획봇 정리 카드 생성
→ 그래도 불명확하면 싱드가 대장에게 질문

배포 위험 발견
→ 운영봇 또는 리뷰봇이 위험 항목 정리
→ 싱드가 대장 승인 여부 확인
```

중요:

- 같은 실패를 반복하면 원인 분석 카드를 따로 만든다.
- “대충 통과”는 금지한다.
- 완료 보고에는 실패했던 항목과 해결 여부를 함께 적는다.

## 9. 처음 적용할 때의 안전 절차

처음부터 큰 기능을 자동으로 돌리지 않는다.

권장 순서:

1. `--preview`로 파이프라인 모양만 확인한다.
2. `--hold`로 카드는 만들되 자동 실행을 멈춘다.
3. 작은 문서 작업 하나로 end-to-end 흐름을 확인한다.
4. 기능 개발은 작은 단위부터 넣는다.
5. 배포 작업은 승인 게이트가 제대로 작동하는지 먼저 확인한다.

미리 보기 예시:

```bash
cd /home/wrhrgw/gw
./scripts/groupware-auto-workflow.sh \
  --preview \
  --type feature \
  "샘플 기능" \
  "자동 루프 구조가 올바른지 미리 확인한다."
```

대기 카드 생성 예시:

```bash
cd /home/wrhrgw/gw
./scripts/groupware-auto-workflow.sh \
  --hold \
  --type feature \
  "샘플 기능" \
  "카드 생성과 의존성 연결만 확인한다."
```

## 10. 싱드의 최종 보고 형식

싱드는 최종 보고를 아래 형식으로 한다.

```text
완료한 일:
- 무엇을 했는지

확인한 근거:
- 어떤 파일/명령/테스트/로그로 확인했는지

문제와 처리:
- 실패했던 항목
- 수정 여부
- 아직 미확인인 항목

대장이 알아야 할 점:
- 위험 요소
- 결정이 필요한 부분

다음 추천 액션:
- 바로 이어서 할 일 1~3개
```

보고는 짧게 하되, **검증 근거와 미확인 사항은 숨기지 않는다**.

## 12. 첫 실전 테스트 후보

권장 첫 테스트는 위험이 낮은 문서 작업이다.

```bash
cd /home/wrhrgw/gw
./scripts/groupware-auto-workflow.sh \
  --type docs \
  --idempotency-key "groupware-loop-first-doc-test" \
  "그룹웨어 봇 루프 첫 실전 테스트" \
  "groupware-bot-loop.md 문서를 기준으로 사용자에게 그룹웨어 봇 루프 사용법을 짧게 안내하는 문서를 작성하고, 실제 운영 문서와 충돌이 없는지 확인한다."
```

이 테스트의 성공 기준:

- Kanban 카드가 생성된다.
- 담당 봇이 작업을 잡는다.
- 문서 결과가 남는다.
- 리뷰 또는 확인 결과가 남는다.
- 싱드가 최종 보고할 수 있는 정보가 모인다.
