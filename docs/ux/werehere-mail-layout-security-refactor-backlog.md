## UI/UX 리팩토링 규칙

### 1. 공식 Flex/Grid 컴포넌트 우선

- Ant Design 또는 shadcn/ui 등 프로젝트에 설치된 UI 라이브러리의 공식 `Flex`/`Grid` 패턴을 우선한다.
- 설치된 UI 라이브러리와 현재 공통 컴포넌트를 먼저 확인한 뒤, 임의 div/margin 조합으로 새 레이아웃을 만들지 않는다.
- 라이브러리 컴포넌트가 없다면 공통 래퍼 컴포넌트와 의미 기반 토큰을 먼저 만든다.

### 2. 상단 검색 필터 고정

- 검색 필터 영역은 Grid 기반으로 구성한다.
- 검색어, 기간, 상태, 보낸사람/받는사람 등 필터 박스는 같은 높이와 균등한 폭을 갖도록 한다.
- 필터 요소가 튀어나오거나 임의 줄바꿈되지 않아야 한다.
- 개별 margin 대신 부모 Grid/Flex의 `gap`과 공통 spacing token을 사용한다.

### 3. 중앙 일괄 처리 바 정렬

- 선택한 메일의 읽음/삭제/이동/스팸/보관 등 일괄 처리 버튼은 하나의 toolbar 컨테이너 안에 둔다.
- toolbar 기본 구조는 `display: flex`, `align-items: center`, `gap: 8px` 기준을 따른다.
- 버튼 높이, 아이콘 정렬, 텍스트 baseline이 검색 필터/상세 툴바와 어긋나지 않게 한다.

### 4. 우측 메일 상세창 툴바 정렬

- 답장, 전달, 삭제, 더보기, 닫기 등 우측 상세창 액션 버튼은 하나의 toolbar 컨테이너 안에 둔다.
- 좌측 목록/중앙 처리 바/우측 상세 툴바의 수평 기준선이 맞는지 확인한다.
- 상세창 내부에서 버튼별 개별 margin으로 위치를 맞추지 않는다.

### 5. 임의 크기·간격 금지

- 새 코드에서 임의 크기와 임의 간격을 창조하지 않는다.
- 기본 간격은 기존 UI/UX 표준 문서에 따라 8px, 16px, 24px 단위를 사용한다.
- 예외가 필요하면 `--mail-*` 또는 `--feature-page-*` 형태의 의미 기반 토큰으로 정의하고 테스트로 잠근다.

## 보안·아키텍처 분리 규칙

### 1. 프론트엔드 인증정보 노출 금지

프론트엔드에는 아래 값이 절대 포함되면 안 된다.

- SMTP host/user/password
- SMTP auth token
- 외부 메일 API key
- 외부 API secret
- provider access token/refresh token
- DNS 인증용 private key
- `.env`, `.secrets` 파일 내용

### 2. 백엔드 환경 변수 격리

- SMTP/API provider 설정은 백엔드 서버 내부 환경 변수에서만 읽는다.
- 프론트엔드는 provider 종류, host, key, secret을 알 필요가 없다.
- 프론트엔드는 메일 작성 데이터와 첨부 id 등 업무 데이터만 API로 전송한다.
- 백엔드는 환경 변수 유무를 감지하고, 미설정이면 명확한 오류 코드로 실패해야 한다.

### 3. 프론트엔드 역할 제한

프론트엔드 역할은 다음으로 제한한다.

- 메일 작성 UI 렌더링
- 수신자/참조/제목/본문/첨부 id 입력 수집
- 내부 API 호출
- 백엔드 오류 코드와 사용자 안내 표시

프론트엔드가 직접 외부 SMTP/API provider를 호출하거나 인증정보를 조립하면 안 된다.

### 4. 백엔드 역할

백엔드는 다음을 담당한다.

- SMTP/API provider 설정 읽기
- 메일 발송 권한 검사
- 내부/외부 수신자 분리
- provider 미설정 시 안전 실패
- 발송 로그 저장
- 실패 코드/메시지 기록
- 감사로그 및 보안 로그 기록

### 5. Provider-ready 단계 분리

- 실제 외부 메일 발송은 별도 승인 전까지 완료로 보고하지 않는다.
- 1차는 provider-ready 기반, 미설정 안전 실패, 로그 구조, 프론트/백엔드 경계 분리를 목표로 한다.
- provider secret 등록, DNS 인증, 실제 외부 수신 확인은 별도 승인 게이트로 유지한다.

## 구현 전 점검 목록

- 현재 설치된 UI 라이브러리 확인
- 메일 화면의 검색 필터, 일괄 처리 바, 상세 툴바 selector/class 확인
- 프론트엔드 bundle/source에서 SMTP/API key 관련 문자열 검색
- API route/server action에서 provider 환경 변수 접근 위치 확인
- `.env`, `.secrets`, Cloudflare secret, GitHub secret 경계 확인
- 기존 외부메일 provider-ready 문서와 충돌 여부 확인

## 검증 기준

### UI 검증

- 상단 검색 필터가 Grid 기반으로 균등 정렬된다.
- 중앙 일괄 처리 바가 toolbar 규격으로 정렬된다.
- 우측 상세창 툴바가 toolbar 규격으로 정렬된다.
- 세 영역의 높이와 수평 기준선이 PC 기준에서 어긋나지 않는다.
- 임의 margin/width/height 값이 새로 추가되지 않는다.

### 보안 검증

- 프론트엔드 source/bundle에 SMTP/API key/secret이 포함되지 않는다.
- provider 설정은 백엔드 환경 변수에서만 읽는다.
- provider 미설정 상태에서는 성공처럼 보이는 응답을 반환하지 않는다.
- 외부메일 발송은 `EXTERNAL_MAIL_NOT_CONFIGURED` 등 명확한 오류와 로그로 안전 실패한다.
- secret 값은 로그, 테스트 출력, PR 본문, 사용자 보고에 노출하지 않는다.

- 실제 외부 메일 발송, provider secret 등록, DNS 변경은 별도 승인 전까지 진행하지 않는다.
