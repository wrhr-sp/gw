# 위아히어 그룹웨어 프론트엔드 UI 표준 전면 전환 지침

## 1. 최상위 기준

본 지침은 위아히어 그룹웨어의 모든 프론트엔드 화면, 컴포넌트,
레이아웃, 디자인 토큰, 상태값 색상, 버튼 규칙, 메뉴 구조, 폼 구조,
테이블 구조에 적용하는 최상위 UI 표준이다.

신규 화면뿐 아니라 기존에 구현된 모든 프론트엔드 화면에도 적용한다.
기존 UI 기준과 본 지침이 충돌하는 경우 본 지침을 우선한다.

기존에 사용하던 화면별 개별 디자인, 임의 컴포넌트, 임의 색상,
임의 버튼 스타일, 임의 레이아웃, 임의 상태값 표시는 본 지침 기준으로
전면 정리한다.

## 2. 기존 UI 기준 폐기 및 재정렬

다음 항목은 모두 본 지침 기준으로 재정렬한다.

- 기존 컴포넌트 구조
- 기존 디자인 토큰
- 기존 레이아웃 토큰
- 기존 상태값 색상 규칙
- 기존 버튼 규칙
- 기존 메뉴 구조
- 기존 목록 화면 구조
- 기존 상세 화면 구조
- 기존 작성 화면 구조
- 기존 폼 구조
- 기존 테이블 구조
- 기존 Confirm Dialog 규칙
- 기존 Audit Log 표시 방식
- 기존 첨부파일 표시 방식
- 기존 결재선 표시 방식

기존 구현 중 재사용 가능한 코드는 폐기하지 않아도 된다. 다만 반드시
본 지침의 공통 컴포넌트 또는 business component 구조에 맞게 역할을
재분류하고 수정한다.

## 3. 오픈소스 기반 UI 스택

그룹웨어 전체 UI 통일을 위해 오픈소스 기반은 다음 조합을 우선 적용한다.

- shadcn-admin: 관리자 화면 골격
- shadcn/ui: 공통 UI 컴포넌트
- Tailwind CSS: 스타일 시스템
- TanStack Table: 목록 테이블
- TanStack Query: API 상태관리
- React Hook Form: 폼 상태관리
- Zod: 입력값 검증
- Tremor: 대시보드/차트 컴포넌트

단, 오픈소스 UI를 그대로 복사하지 않는다. 접근성 구조, 상태관리 방식,
레이아웃 패턴만 참고하고 위아히어 그룹웨어 디자인 시스템으로 재구성한다.

AGPL 또는 라이선스가 불명확한 코드는 본체로 흡수하지 않는다.

## 4. 디자인 시스템 파일 기준

상태값 색상, 버튼 규칙, 메뉴 구조, 디자인 토큰은 전역 파일에서 관리한다.

- `apps/web/design-system/status-colors.ts`
- `apps/web/design-system/button-rules.ts`
- `apps/web/design-system/menu-config.ts`
- `apps/web/design-system/tokens.ts`

앞으로 개별 화면에서 임의 색상, 임의 여백, 임의 버튼 스타일,
임의 상태 배지 스타일을 직접 정의하지 않는다.

모든 화면은 `design-system` 하위 전역 규칙을 우선 사용한다.

## 5. 공통 컴포넌트 우선 구현

다음 공통 컴포넌트를 먼저 구현하고, 모든 화면에서 우선 재사용한다.

- PageHeader
- FilterBar
- DataTable
- StatusBadge
- ConfirmDialog
- EmptyState
- SummaryCard
- DetailSection
- FormSection
- ActionButtonGroup
- AttachmentPanel
- AuditLogPanel
- ApprovalLinePanel

신규 화면을 만들 때는 먼저 위 공통 컴포넌트로 구현 가능한지 확인한다.
공통 컴포넌트로 처리하기 어려운 업무 특화 UI가 필요한 경우에만
business component를 추가한다.

## 6. 화면 유형별 표준 패턴

모든 목록 화면은 다음 패턴을 따른다.

```text
PageHeader
→ FilterBar
→ DataTable
→ Pagination
```

모든 상세 화면은 다음 패턴을 따른다.

```text
PageHeader
→ SummaryCard
→ DetailSection
→ AttachmentPanel
→ AuditLogPanel
```

결재 또는 승인 관련 상세 화면은 필요 시 다음 컴포넌트를 추가한다.

```text
→ ApprovalLinePanel
```

모든 작성 화면은 다음 패턴을 따른다.

```text
PageHeader
→ FormSection
→ ActionButtonGroup
→ ConfirmDialog
```

위 패턴과 다른 화면 구성이 필요한 경우에는 임의 구현하지 말고,
예외 사유를 보고하고 승인받은 뒤 적용한다.

## 7. 계정관리 화면 기준 샘플

계정관리 화면을 위아히어 그룹웨어 UI 표준의 기준 샘플 화면으로 먼저
완성한다.

계정관리 화면에는 다음 요소를 포함한다.

- 목록 화면
- 검색/필터
- 상세 화면
- 계정 생성 Wizard
- 계정 수정
- 상태 변경
- 권한 템플릿
- 접근범위
- 로그인 보안
- 감사로그

계정관리 화면이 표준 패턴으로 완성된 뒤, 전자결재, 업무관리, 문서관리,
사업장관리, 거래처관리, 설정, 대시보드 화면은 동일한 UI 패턴을 재사용한다.

## 8. 우측 상세패널과 상단바 팝업 레이어

우측 상세패널은 사원생성패널을 공통 기준으로 삼는다. 패널은 상단바를 덮지 않고 상단바 바로 아래에서 시작해야 하며, `position: fixed`, `top: var(--employee-detail-panel-top)`, `height: calc(100dvh - var(--employee-detail-panel-top))`, `grid-template-rows: auto auto minmax(0, 1fr)` 구조를 사용한다. 패널 전체가 아니라 본문 영역만 `overflow-y: auto`로 스크롤한다.

상단바 버튼에서 열리는 프로필 팝업, 포털 전환 팝업, 통합설정/공지/알림 모달은 우측 상세패널과 일반 모달보다 높은 전용 z-index 토큰을 사용해 항상 최상단에 표시한다. 화면별 임의 z-index 숫자를 직접 쓰지 않는다.

## 9. ZITADEL 연동 원칙

프론트엔드에서는 ZITADEL API를 직접 호출하지 않는다.
프론트엔드는 위아히어 백엔드 API만 호출한다.

ZITADEL 사용자 생성, 권한 매핑, 로그인 정책, 감사로그 기록은 백엔드에서
처리한다.

프론트엔드는 ZITADEL의 기술 개념을 그대로 노출하지 않고, 위아히어 업무
기준의 계정관리 화면으로 표현한다.

## 10. 앞으로의 개발 원칙

앞으로 개발되는 모든 프론트엔드 화면은 본 지침을 무조건 따른다.

- 개발자는 화면을 개별적으로 새로 디자인하지 않는다.
- 개발자는 기존 스타일을 임의로 유지하지 않는다.
- 개발자는 화면별 임의 컴포넌트를 먼저 만들지 않는다.
- 개발자는 공통 컴포넌트, 디자인 시스템 토큰, 표준 레이아웃 패턴을
  우선 사용한다.
- 본 지침과 충돌하는 기존 코드가 발견되면, 해당 코드는 본 지침 기준으로
  수정한다.

## 11. 예외 처리

예외 구현이 필요한 경우에는 사전에 다음 내용을 보고한다.

- 예외가 필요한 화면 또는 컴포넌트
- 기존 표준으로 처리할 수 없는 이유
- 제안하는 예외 구현 방식
- 다른 화면에 미치는 영향
- 추후 공통 컴포넌트로 전환 가능 여부

승인되지 않은 예외 구현은 허용하지 않는다.

## 12. 적용 순서

1. 본 지침과 디자인 시스템 파일을 기준선으로 고정한다.
2. 공통 컴포넌트 골격을 구현한다.
3. 계정관리 화면을 기준 샘플로 먼저 표준화한다.
4. 전자결재, 업무관리, 문서관리, 사업장관리, 거래처관리, 설정,
   대시보드를 같은 패턴으로 확장한다.
5. 기존 화면은 기능 단위 PR과 회귀 테스트를 붙여 전환한다.

## 13. 별도 승인 게이트

다음 항목은 본 UI 표준 전환과 별도 승인 대상으로 유지한다.

- production DB 실데이터 변경
- 비밀값 입력 또는 교체
- DNS/custom domain 변경
- 유료 리소스 생성
- 외부 SMTP/API/provider 실제 연동
- destructive migration
