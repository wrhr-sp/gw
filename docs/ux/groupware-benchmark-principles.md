# 그룹웨어 UX 벤치마크 원칙

## 1. 문서 목적

이 문서는 국내 그룹웨어/HR/근태/전자결재/문서/게시판 서비스의 공개 제품 소개 페이지와 공개 도움말에서 읽을 수 있는 패턴을 안전하게 추상화해, 앞으로 우리 서비스가 따라야 할 UI/UX 기준을 정리한 문서입니다.

중요한 전제는 아래와 같습니다.

- 공개 페이지, 공개 도움말, 공개 제품 소개만 참고한다.
- 로그인 뒤 화면, 유료 데모 내부 화면, 비공개 캡처는 참고하지 않는다.
- 특정 서비스의 문구, 색상, 로고, 레이아웃을 복제하지 않는다.
- 우리가 가져오는 것은 화면 복제가 아니라 정보구조와 업무 우선순위 같은 추상 원칙이다.

이 문서는 Phase 6 모바일/PWA 후속 작업과 Phase 7+ 기능 카드가 공통으로 참조하는 기준 문서다.

## 1-1. 개발 handoff 한눈에 보기

다음 UI 작업자는 아래 7가지만 먼저 기억하면 된다.

1. 메뉴를 새로 만들기 전에 먼저 기존 업무 묶음(근태/휴가, 전자결재, 게시판/문서, 관리자) 안에 넣을 수 있는지 본다.
2. 넓은 화면은 왼쪽 사이드바, 좁은 화면은 하단 탭 기본안을 유지한다.
3. 관리자 기능은 일반 사용자 하단 탭에 섞지 않는다.
4. 상태를 바꾸는 CTA 는 카드 맨 아래가 아니라, 관련 상태 요약 바로 옆이나 바로 아래에 둔다.
5. 빈 상태/오류/오프라인 문구는 "왜 이런 상태인지 + 지금 사용자가 할 일" 순서로 쓴다.
6. preview URL 이 있어도 manifest 와 API 기본 경로는 same-origin 상대 경로(`/manifest.webmanifest`, `/api/*`)를 유지한다. 다만 공개 preview 재배포/build 상태에 따라 새 `/api/*` 결과가 아직 다시 검증되지 않았을 수 있다는 운영 한계와, 제품 기본 원칙은 같은 말이 아니다.
7. Phase 7 급여/노무 UI 는 근태/휴가와 가까운 묶음으로 시작하되, 법적 민감도 안내와 관리자 경계를 더 분명하게 보여 준다.

바꿔 말하면, 후속 카드가 참고해야 하는 핵심은 "업무 묶음 중심 IA, 오늘 처리할 일 우선, 화면 크기별 탐색 껍데기만 변경, same-origin 기본값 유지, 관리자 경계 명확화"다.

## 2. 공개 참고 출처와 안전한 해석 범위

2026-06-10 기준, 아래 공개 페이지를 확인했다.

1. Dooray 공개 홈페이지 `https://dooray.com/`
   - 공개 내비게이션에서 프로젝트, 메신저, 메일, 화상회의, 드라이브, 위키, 캘린더, 주소록, 결재, 홈/게시판, 자원예약, 폼 같은 업무 묶음을 노출한다.
   - 안전하게 가져올 수 있는 점: 그룹웨어는 기능을 "단일 목록"이 아니라 업무 묶음 단위로 보여 주는 편이 자연스럽다.

2. 하이웍스 공개 제품 페이지 `https://www.hiworks.com/product/function/groupware`
   - 공개 메타 설명에서 기업메일, 전자결재, 근무관리, 기업메신저, 전자세금계산서 같은 대표 기능군을 함께 설명한다.
   - 안전하게 가져올 수 있는 점: 한국형 그룹웨어는 메일/결재/근태/협업이 한 제품 경험 안에서 이어진다는 기대가 강하다.

3. 이카운트 공개 홈페이지 `https://www.ecount.com/kr/`
   - 공개 페이지 텍스트에서 근태관리, 인사관리, 그룹웨어, 게시판, 전자결재, 클라우드 스토리지, 일정관리, 출퇴근관리, 웹메일 같은 업무 항목이 함께 보인다.
   - 안전하게 가져올 수 있는 점: 운영/관리 업무와 일상 협업 업무가 분리되되 한 제품 안에서 이동 가능한 구조가 필요하다.

4. LINE WORKS 공개 소개 페이지 `https://line.worksmobile.com/kr/`
   - 공개 설명에서 채팅, 메일, 캘린더, 파일 관리 같은 그룹웨어 기능을 PC뿐 아니라 스마트폰에서도 쓸 수 있다고 안내한다.
   - 안전하게 가져올 수 있는 점: 모바일은 축소판이 아니라 핵심 업무를 끝낼 수 있는 우선순위 재배치가 중요하다.

5. 더존 공개 사이트 `https://www.douzone.com/Amaranth10`
   - 공개 사이트 분류에서 인사/급여/근태관리, 업무포털, 그룹웨어, 협업/프로젝트관리 같은 업무 영역 구분이 드러난다.
   - 안전하게 가져올 수 있는 점: HR/근태/급여와 협업/문서/결재는 메뉴 구조상 구분하되, 사용자는 같은 업무 허브 안에서 이어서 움직일 수 있어야 한다.

정리하면, 공개 정보만으로도 아래 패턴은 충분히 읽힌다.

- 기능은 "업무 묶음"으로 제시된다.
- 홈/포털은 오늘 처리할 일 중심이다.
- 모바일은 별도 앱 흉내보다 핵심 업무 우선순위 재정렬이 중요하다.
- 관리자 기능과 일반 사용자 기능은 분명히 나뉜다.

## 3. 우리 서비스가 채택할 핵심 UX 원칙

### 원칙 A. 메뉴는 기능 나열보다 업무 묶음 중심으로 구성한다.

권장 1차 묶음은 아래와 같다.

- 홈/대시보드
- 근태/휴가
- 전자결재
- 게시판/공지
- 문서/파일
- 조직/직원
- 관리자

세부 원칙:

- 상위 메뉴는 사용자가 "무슨 일을 하려는지" 기준으로 보여 준다.
- 게시판과 공지는 완전히 별도 세계가 아니라 같은 커뮤니케이션 묶음 안에서 관리한다.
- 문서와 파일은 협업 문맥을 공유하되, 권한과 보관 규칙은 별도 경계로 설명한다.
- 급여/노무가 붙는 Phase 7+에서는 근태/휴가와 가까운 묶음으로 연결하되, 법적 민감도 때문에 별도 강조 구간을 둔다.

### 원칙 B. 홈/대시보드는 "오늘 처리할 일"이 먼저다.

홈에서 우선 노출할 정보 순서는 아래를 기본값으로 둔다.

1. 지금 바로 해야 하는 상태 변경 액션
2. 오늘 확인할 승인/대기/예외
3. 최근 기록 요약
4. 자주 가는 업무 진입점
5. 안내/정책/참고 링크

적용 예시:

- 근태: 출근/퇴근, 마지막 기록, 정정 요청
- 휴가: 잔여, 신청, 승인 대기
- 전자결재: 내 승인함, 내 기안함, 반려/보완 필요 건
- 게시판/문서: 새 공지, 읽지 않은 글, 최근 문서 공간

홈은 모든 정보를 다 보여 주는 곳이 아니라, 사용자가 가장 짧은 경로로 다음 행동을 고를 수 있게 하는 곳으로 정의한다.

### 원칙 C. 내비게이션은 데스크톱과 모바일의 역할을 분리한다.

현재 승인된 기본안은 아래와 같다.

- 넓은 화면: 왼쪽 사이드바 기본
- 좁은 화면: 하단 탭 유지

세부 원칙:

- 데스크톱/태블릿 가로 폭에서는 왼쪽 사이드바가 1차 탐색 기준이다.
- 모바일에서는 한 손 사용을 고려해 하단 탭으로 핵심 4~6개 섹션에 빠르게 이동한다.
- 모바일에서 하단 탭에 다 담기지 않는 항목은 "더보기" 또는 화면 내부 섹션 링크로 푼다.
- 데스크톱과 모바일이 서로 다른 정보구조를 갖지 않게, 같은 route 체계를 다른 탐색 껍데기로 보여 준다.

즉, IA는 하나로 유지하고, 탐색 UI만 화면 크기에 따라 달라진다.

### 원칙 D. CTA는 카드 아래가 아니라 "결정 직전"에 둔다.

기본 배치 기준:

- 상태 변경 CTA는 관련 상태 요약 바로 아래 또는 같은 카드 안에 둔다.
- 승인/반려처럼 실수 비용이 큰 CTA는 상태 정보, 대상 정보, 경고 문구 뒤에 배치한다.
- 목록 화면 맨 아래에만 CTA를 두어 다시 스크롤하게 만들지 않는다.
- 모바일에서는 엄지 도달 범위를 고려해 주요 CTA를 하단 또는 카드 하단 가까이에 두되, 의미 없는 고정 버튼 남발은 피한다.

도메인별 기준:

- 근태: 출근/퇴근 CTA와 마지막 기록을 붙여 둔다.
- 휴가: 잔여와 신청 CTA를 같은 맥락에서 본다.
- 결재: 문서 상태, 결재선, 영향 설명 뒤에 승인/반려 CTA를 둔다.
- 게시판/문서: 읽기와 관리 액션을 분리하고, 작성/업로드는 권한이 있을 때만 분명히 드러낸다.

### 원칙 E. 관리자와 일반 사용자 경계는 화면부터 분명해야 한다.

권장 규칙:

- 일반 사용자는 자기 업무 중심 메뉴를 먼저 본다.
- 관리자 메뉴는 별도 섹션 또는 별도 사이드바 그룹으로 분리한다.
- 관리자 전용 화면은 "운영 설정"과 "사용자 일상 업무"를 섞지 않는다.
- 권한이 없는 액션은 숨기거나 비활성화하되, 왜 안 되는지 설명 가능한 문구를 남긴다.

우리 저장소 기준으로는 아래 경계를 유지한다.

- 일반 사용자 업무: `/dashboard`, `/attendance`, `/leave`, `/approvals`, `/boards`, `/documents`
- 관리자 업무: `/admin`, `/admin/users`, `/admin/policies`, `/admin/audit-logs`

### 원칙 F. 기능은 단건 화면보다 흐름 단위로 묶는다.

1. 근태/휴가 묶음
   - 오늘 상태
   - 출퇴근/정정
   - 휴가 잔여/신청
   - 승인 대기/처리 결과

2. 전자결재 묶음
   - 내 승인함
   - 내 기안함
   - 문서 상세
   - 승인/반려/보완 요청

3. 게시판/문서 묶음
   - 공지/게시판 목록
   - 읽지 않은 항목
   - 게시글 상세/댓글
   - 문서 공간/첨부 metadata

이 원칙은 route 를 줄이라는 뜻이 아니라, 사용자가 "다음 단계"를 예상할 수 있게 화면 연결을 설계하라는 뜻이다.

### 원칙 G. 빈 상태, 오류, 오프라인 안내는 낙관적 문구보다 제약 설명을 우선한다.

기본 기준:

- 데이터 없음: 왜 비어 있는지와 다음 행동을 같이 보여 준다.
- 권한 없음: 숨김만 하지 말고 관리자/정책 사유를 설명한다.
- 오류: 재시도, 확인할 값, 대체 경로를 알려 준다.
- 오프라인: 읽기 가능 범위와 막히는 액션을 분리해 적는다.

특히 금지할 것:

- 근태 등록, 휴가 승인, 결재 승인처럼 상태 변경이 필요한 일을 오프라인에서도 성공처럼 보이게 만드는 것
- placeholder 단계인데 실제 운영 완료처럼 느껴지게 하는 문구
- 오류 원인을 알 수 없는데 "잠시 후 다시"만 반복하는 안내

### 원칙 H. 모바일은 축소판이 아니라 우선순위 재정렬 버전이어야 한다.

모바일에서 먼저 보여 줄 것은 아래다.

- 오늘 가장 많이 누르는 액션
- 지금 상태를 이해하는 최소 정보
- 다음 단계로 가는 짧은 링크
- 읽기/처리 구분이 분명한 카드 구조

모바일에서 뒤로 미루는 것은 아래다.

- 긴 표 전체 노출
- 드문 관리자 설정
- 고급 필터 전체 세트
- 부가 설명 위주의 홍보성 블록

즉, 모바일은 "같은 기능의 작은 화면"이 아니라 "핵심 행동부터 다시 배열한 화면"이어야 한다.

### 원칙 I. PWA/preview/API 연결 원칙은 UX 문서에서도 흔들리지 않아야 한다.

앞으로도 아래 기준을 유지한다.

- manifest 는 상대 경로 유지
- API 기본 경로는 same-origin `/api/*` 우선 유지
- preview URL 이 생겨도 절대 preview 도메인을 문서/코드 기본값으로 박아 넣지 않는다.
- 모바일 UX 개선이 API origin 분리나 보안 경계 완화로 이어지지 않게 한다.

이 원칙은 디자인 원칙이면서 동시에 운영 안전장치다.

## 4. 현재 저장소에 바로 적용하는 IA/탐색 기준

현재 route 기준 권장 구조는 아래다.

### 공통 IA

- 홈: `/`
- 대시보드: `/dashboard`
- 근태: `/attendance`
- 휴가: `/leave`
- 전자결재: `/approvals`
- 게시판: `/boards`
- 문서함: `/documents`
- 로그인: `/login`
- 오프라인 안내: `/offline`
- 관리자 묶음: `/admin/*`

### 넓은 화면

왼쪽 사이드바 1차 그룹 예시:

- 홈은 최상단 단독 항목으로 두고, 어떤 묶음에도 넣지 않는다.
- 묶음 제목 아래 설명 문구는 사이드바에서 노출하지 않는다.

1. 협업/소통
   - 메일
   - 메신저
   - 알림
   - 게시판
   - 주소록
   - 직원
   - 익명게시판(준비중)
2. 일정/개인 업무
   - 공통 업무
   - 캘린더(준비중)
   - 예약(준비중: 회의실·차량·공용장비 같은 사내 자원 예약 후보이며, 기능 구현은 후속 작업)
   - 보고(준비중)
   - ToDO+(준비중)
3. 근무/인사
   - 근태
   - 휴가
   - 인사 업무
   - 직원교육(준비중)
   - 고용전자계약(준비중)
4. 결재/문서
   - 전자결재
   - 전사 문서함
   - 자료실(준비중)
   - 드라이브(준비중)
5. 급여/비용
   - 급여(본인 급여명세서 조회)
   - 경비(준비중)
6. 운영/기타
   - 차량운행일지(준비중)

실제 route가 없는 준비중 항목은 깨진 사이드바 링크로 만들지 않고 비활성 자리 확보 상태로 둔다.

### 좁은 화면

하단 탭 기본안:

- 홈
- 근태
- 휴가
- 전자결재
- 게시판
- 문서함

주의:

- 탭 수가 과하면 핵심 4~5개만 남기고 나머지는 화면 내부 진입점으로 보낸다.
- 관리자 메뉴는 하단 탭 기본항목으로 올리지 않는다.

## 5. 도메인별 화면 우선순위 기준

### 근태/휴가

먼저 보여 줄 것:

- 오늘 상태
- 출근/퇴근 CTA
- 마지막 기록
- 잔여 휴가
- 신청/정정 진입

뒤로 미룰 것:

- 긴 월간 표 전체
- 고급 필터 전체
- 관리자 일괄 처리

### 전자결재

먼저 보여 줄 것:

- 대기 건수
- 내 승인함
- 내 기안함
- 문서 상태
- 승인/반려 CTA

뒤로 미룰 것:

- 부가 메타데이터 전체
- 드문 참조/합의 세부 옵션
- 법적 고도화 기능 소개

### 게시판/문서

먼저 보여 줄 것:

- 새 공지/읽지 않음
- 최근 게시글/문서 공간
- 제목/상태/작성자/수정일 같은 최소 메타데이터
- 읽기와 작성 액션의 분리

뒤로 미룰 것:

- 복잡한 정렬/검색 세트
- 미리보기 고도화
- 외부 공유/다운로드 운영 기능

## 6. Phase 7+ 작업자가 반드시 지켜야 할 구현 체크리스트

1. 새 메뉴를 추가할 때는 먼저 기존 업무 묶음 안에 넣을 수 있는지 검토한다.
2. 급여/노무 기능은 근태와 가까이 배치하되, 법적 민감도 안내를 별도 강조한다.
3. 넓은 화면은 왼쪽 사이드바, 좁은 화면은 하단 탭이라는 기본안을 깨지 않는다.
4. 관리자 기능을 일반 사용자 하단 탭에 섞지 않는다.
5. 상태 변경 CTA는 관련 상태 요약과 가까운 위치에 둔다.
6. 빈 상태/오류/오프라인 문구는 "제약 설명 + 다음 행동" 구조로 작성한다.
7. 모바일에서도 same-origin `/api/*` 와 권한 경계가 흐려지지 않는지 같이 확인한다.
8. 공개 경쟁사 UI를 그대로 닮게 만드는 문구/배치/스타일 복제는 피한다.

## 7. 미정 사항과 승인 필요 사항

### 미정 사항

- 모바일 하단 탭을 5개로 줄일지 6개까지 허용할지
- 게시판과 문서함을 모바일에서 각각 독립 탭으로 둘지, 협업 묶음으로 합칠지
- 급여/노무가 붙는 Phase 7 에서 근태/휴가와 같은 묶음 안에 둘지, 별도 상위 메뉴로 뺄지

### 승인 필요 사항

- 관리자 IA 를 일반 사용자 IA 와 완전히 분리하는 큰 구조 변경
- PWA 설치 이후 네이티브 앱 수준 경험처럼 보이게 만드는 push/background sync 확대
- preview/production 외부 공개 범위 확대
- 실제 급여 계산, 세무/노무 법정 처리처럼 전문가 검수가 필요한 UX 확정

## 8. 이 문서를 먼저 참조해야 하는 후속 작업

아래 작업은 시작 전에 이 문서를 먼저 본다.

- Phase 6 모바일 CTA/accessibility remediation
- Phase 7 급여/노무 정보구조 설계
- 게시판/문서 모바일 재배치
- 대시보드 quick action 재정렬
- 관리자/일반 사용자 권한 경계 UX 정리
- Cloudflare preview 이후 responsive QA 체크리스트 작성

## 9. 한 줄 결론

우리는 국내 그룹웨어 공개 패턴을 참고하되, 화면 복제가 아니라 "업무 묶음 중심 IA + 오늘 처리할 일 우선 + 넓은 화면 왼쪽 사이드바 + 모바일 하단 탭 + 명확한 권한 경계 + same-origin 운영 경계 유지"를 우리 서비스의 기본 UX 규칙으로 채택한다.
