# 그룹웨어 UI/UX 표준화 백로그

## 한 줄 요약

그룹웨어 UI/UX 표준화 백로그는 `docs/ux/werehere-frontend-ui-standard.md`를 최상위 기준으로 두고, 계정관리 화면을 기준 샘플로 먼저 완성한 뒤 모든 기존/신규 화면을 공통 컴포넌트·디자인 시스템·표준 패턴으로 전환하기 위한 실행 백로그다.

## 현재 상태

- 문서화 일자: 2026-07-08
- 상태: 최상위 UI 표준 전면 전환 기준 반영
- 우선순위: 계정관리 화면을 기준 샘플로 먼저 완성하고, 이후 전자결재·업무관리·문서관리·사업장관리·거래처관리·설정·대시보드에 같은 패턴을 재사용한다.
- 최상위 기준: `docs/ux/werehere-frontend-ui-standard.md`

## 배경

현재 그룹웨어 화면은 기능별로 입력창, 버튼, 필터, 표, 여백, 좌우 분할 레이아웃의 규격이 서로 달라 가독성과 유지보수성이 떨어질 위험이 있다. 앞으로 개발하는 모든 그룹웨어 화면은 공통 UI 규격을 우선 적용해야 하며, 기존 화면도 기능 단위로 회귀 방지 테스트를 붙여 정리한다.

## 전역 UI/UX 강제 원칙

### 1. 기성 UI 라이브러리 규격 우선

- `Input`, `Select`, `DatePicker`, `Button`, `Table` 계열 컴포넌트는 프로젝트에 설치된 UI 라이브러리의 공식 가이드 규격을 우선한다.
- 임의 크기, 임의 여백, 페이지별 개별 스타일을 새로 만들지 않는다.
- 라이브러리 컴포넌트가 아직 없거나 기존 구조와 충돌하면, 먼저 공통 래퍼/토큰을 만들고 기능 페이지는 그 래퍼를 사용한다.
- 라이브러리 전환은 동작 회귀가 큰 작업이므로, 저장·수정·삭제·신청·승인·반려 같은 mutation 흐름은 반드시 API/DB 재조회 검증과 함께 처리한다.

### 2. 상단 검색/필터 바 고정 규격

- 페이지 상단 검색 조건 영역은 Grid 레이아웃을 사용한다.
- 모든 필터 박스의 높이와 너비가 균등하게 보이도록 `repeat(auto-fit, minmax(...))` 또는 고정된 공통 grid token을 사용한다.
- 필터 요소가 삐져나오거나 임의 줄바꿈되지 않게 한다.
- 검색/필터 영역에는 임의 margin보다 부모 `gap`을 우선 사용한다.

### 3. 버튼 툴바 정렬 규격

- 목록 상단, 상세창 내부, 폼 하단의 액션 버튼은 반드시 부모 툴바 컨테이너 안에 둔다.
- 기본 구조는 다음 기준을 따른다.
  - `display: flex`
  - `align-items: center`
  - `gap: 8px`
  - 필요 시 `flex-wrap: wrap`
- 버튼만 따로 absolute 배치하거나 개별 margin으로 맞추지 않는다.

### 4. 여백 표준화

- 화면 안 콘텐츠 박스, 필터, 폼, 리스트, 버튼 묶음은 8px 단위 격자를 우선한다.
- 기본 간격 후보는 `8px`, `16px`, `24px`다.
- `7px`, `13px`처럼 의미 없는 임의 픽셀값은 새 코드에서 금지한다.
- 기존 코드의 임의 픽셀값은 한 번에 전역 치환하지 않고, 영향 범위가 확인된 기능 단위로 토큰화한다.

### 5. 좌우 스플릿 뷰 표준화

- 왼쪽 목록/메뉴와 오른쪽 상세 보기 구조는 Grid 또는 Flex 기반의 견고한 레이아웃을 사용한다.
- 브라우저 크기가 바뀌어도 큰 콘텐츠 박스가 깨지지 않도록 높이와 overflow 체인을 확인한다.
- 리스트 영역은 `width: 100%`와 `min-width: 0`을 기본으로 한다.
- 내부 스크롤은 실제 목록/본문 컨테이너에만 둔다.

## 리팩토링 대상 분류

### 1차: 계정관리 기준 샘플 완성

계정관리 화면은 그룹웨어 UI 표준의 기준 샘플이다. 아래 요소를 먼저 표준 패턴으로 완성한다.

- 목록 화면
- 검색/필터
- 상세 화면
- 계정 생성 Wizard
- 계정 수정
- 상태 변경
- 권한 템플릿
- 접근범위
- 로그인 보안
- 감사로그

### 2차: 기존 핵심 협업 화면 정리

지점관리포털 작업 후 기존 화면은 다음 순서로 정리한다.

1. 메일
   - 메일쓰기 폼
   - 받는사람/참조 입력 영역
   - 메일함 목록
   - 검색/필터 바
   - 중앙 일괄 처리 바
   - 메일 상세/답장/전달 액션 툴바
   - SMTP/API 인증정보 프론트엔드 노출 제거 및 백엔드 환경 변수 격리
   - 세부 기준: `docs/ux/werehere-mail-layout-security-refactor-backlog.md`

2. 메신저
   - 대화목록/채팅방 스플릿
   - 새 메시지 팝업
   - 메시지 검색/참여자 관리
   - 채팅 입력 툴바
   - 첨부/이모티콘/전송 버튼 정렬

3. 게시판
   - 게시판 목록/게시글 목록 2분할
   - 글쓰기 폼
   - 상단 필터/탭
   - 게시글 상세 액션 버튼
   - 댓글/읽음 확인 영역

### 3차: 나머지 업무 화면 확장

- 근태
- 휴가
- 전자결재
- 문서함
- 직원/조직
- 급여/내 급여
- 관리자 설정
- 부서업무포털 화면

## 구현 전 점검 목록

- 현재 설치된 UI 라이브러리와 사용 가능한 공식 컴포넌트를 확인한다.
- `apps/web/app/globals.css`에서 임의 픽셀값, 개별 margin, 페이지별 input/button/table 스타일을 스캔한다.
- `apps/web/content-spacing-boundary.test.tsx` 또는 별도 UI boundary 테스트에 전역 규칙을 추가한다.
- 기능별 변경 전후로 focused render test, typecheck, `build:cf`를 실행한다.
- 배포가 포함된 작업은 live route와 CSS asset marker를 확인한다.

## 완료 기준

- 신규 화면은 공통 UI 규격으로 구현되어야 한다.
- 기존 화면 리팩토링은 기능 단위 PR로 진행한다.
- 필터 바, 버튼 툴바, 폼 컨트롤, 테이블/리스트, 좌우 스플릿의 공통 규칙이 테스트로 잠겨야 한다.
- 화면 동작, API, 권한, DB 저장·조회 흐름은 UI 정리 때문에 깨지면 안 된다.
- 지점관리포털 작업이 끝난 뒤 이 문서를 기준으로 UI/UX 표준화 후속 작업을 재개한다.

## 별도 승인 게이트

아래 항목은 UI 표준화 작업과 별도 승인 대상으로 유지한다.

- production DB 실데이터 변경
- 비밀값 입력/교체
- DNS/custom domain 변경
- 유료 리소스 생성
- 외부 SMTP/API/provider 실제 연동
- destructive migration
