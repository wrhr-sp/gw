# 관리자설정 접근권한/관리자 권한 탭 분리 범위

## 1. 한 줄 정의

이번 1차 목표는 통합설정 안의 `관리자설정` 화면에서
"무엇을 볼 수 있는가"와 "무엇을 바꿀 수 있는가"를 섞지 않도록
`접근권한` 탭과 `관리자 권한` 탭을 preview/state 기준 UI로 분리하는 것입니다.

쉽게 말하면:
- 접근권한 = 메뉴/화면/데이터를 열람하거나 들어갈 수 있는 권한
- 관리자 권한 = 사용자/조직/권한/보안/감사 같은 운영 기능을 변경하거나 부여할 수 있는 권한

이번 단계도 실제 저장, 서버 API, DB, migration, audit log 적재, 운영 반영은 하지 않습니다.

## 2. 왜 지금 이 분리가 필요한가

현재 `apps/web/app/_components/mobile-app-shell.tsx` 기준으로는
통합설정 상단에 `기본 설정` / `관리자설정` 두 탭이 있고,
`관리자설정` 안에는 `관리자 권한 확인` 카드 1개와 토글 4개가 한 묶음으로 들어 있습니다.

현재 토글:
- 조직 조회 모델
- 사용자 역할 정책
- 기본 랜딩 보호
- 감사 로그 미리보기

이 구조는 "조회/열람"과 "운영 변경/정책 부여"를 한 카드에서 같이 보여 주기 때문에,
다음 같은 오해가 생기기 쉽습니다.

1. 감사 로그를 볼 수 있는 권한과 감사 정책을 바꾸는 권한이 같은 뜻처럼 보일 수 있다.
2. 조직/직원 조회 접근과 사용자 역할 변경 권한이 같은 레벨로 읽힐 수 있다.
3. 사용자는 "보는 권한"을 켰는지, "운영 권한"을 켰는지 즉시 구분하기 어렵다.
4. 후속 서버/API 설계에서 read capability 와 write capability 를 분리하기 더 어려워진다.

즉 이번 카드는 새 권한 체계를 만드는 단계가 아니라,
이미 있는 preview 관리자설정 UI를 더 오해 없이 읽히게 다시 묶는 단계입니다.

## 3. 이번에 다시 확인한 현재 사실

확인한 근거:
- `apps/web/app/_components/mobile-app-shell.tsx`
- `apps/web/mobile-app-shell-admin-boundary.test.tsx`
- `docs/guides/unified-settings-secondary-password-preview-flow.md`
- `docs/ux/groupware-benchmark-principles.md`
- `docs/product/groupware-vision-roadmap.md`
- `docs/architecture/admin-role-permission-model-pass-1-scope.md`

현재 저장소 기준 사실:

### 3-1. 통합설정의 1차 상단 탭 구조는 이미 고정돼 있다.
- `기본 설정`
- `관리자설정`

이 상단 2탭 구조는 유지한다.
이번 카드에서 새로 나누려는 것은 `관리자설정` 내부 내용이다.

### 3-2. 설정 진입 전 2차 비밀번호 게이트 기준도 유지해야 한다.
- 설정 관련 진입점에서 먼저 4칸 PIN 게이트를 쓴다.
- 인증 후에도 통합설정은 항상 `기본 설정` 탭부터 열린다.
- 관리자설정 우상단의 폐기된 `2차 비밀번호 변경` 버튼은 되살리지 않는다.
- `내정보 설정` 안 `2차 비밀번호 변경하기` 위치를 유지한다.

즉 이번 UI 분리는 2차 비밀번호 게이트 흐름을 바꾸는 작업이 아니다.

### 3-3. 현재 관리자설정은 local state preview 단계다.
- 현재 권한 상태는 서버 저장이 아니라 프론트 state 로만 움직인다.
- 토글/선택 결과는 preview 용 표시다.
- 새로고침/세션 변경/다른 기기 일관성은 아직 없다.

따라서 이번 카드의 결과는 "권한 구조를 실제로 저장한다"가 아니라
"권한 구조를 화면에서 더 명확하게 구분해 읽게 만든다"다.

## 4. 이번 1차에서 고정할 핵심 결정

### 결정 A. 상단 탭은 그대로 두고, `관리자설정` 안에 하위 2탭을 둔다.

유지:
- `기본 설정`
- `관리자설정`

신규 분리 대상 (`관리자설정` 내부):
- `접근권한`
- `관리자 권한`

즉 사용자는
1. 통합설정 진입
2. 기본적으로 `기본 설정` 탭 확인
3. 필요 시 `관리자설정` 탭 이동
4. 그 안에서 `접근권한` / `관리자 권한` 의미를 다시 구분
순서로 보게 된다.

### 결정 B. `접근권한` 탭은 "볼 수 있음 / 들어갈 수 있음" 중심으로 쓴다.

이 탭은 아래 질문에 답해야 한다.
- 어떤 화면에 들어갈 수 있는가?
- 어떤 데이터/요약을 볼 수 있는가?
- 어떤 관리자 read-only 흐름을 열람할 수 있는가?

권장 항목 묶음:
1. 관리자 홈/설정 진입
2. 조직도/직원 정보 조회
3. 정책/운영 현황 조회
4. 감사 로그 열람

권장 문장 규칙:
- "열람"
- "조회"
- "들어가기"
- "볼 수 있음"

피해야 할 문장:
- "변경"
- "부여"
- "관리"
- "실행"

### 결정 C. `관리자 권한` 탭은 "바꿀 수 있음 / 부여할 수 있음" 중심으로 쓴다.

이 탭은 아래 질문에 답해야 한다.
- 누구의 권한을 바꿀 수 있는가?
- 어떤 운영 정책을 편집할 수 있는가?
- 어떤 보안/감사/권한 작업을 실행할 수 있는가?

권장 항목 묶음:
1. 사용자 역할/상태 변경
2. 조직/부서/권한 정책 관리
3. 기본 랜딩/접근 정책 관리
4. 보안/감사 운영 관리

권장 문장 규칙:
- "변경"
- "부여"
- "관리"
- "편집"
- "실행"

피해야 할 문장:
- 단순 "조회"만으로 끝나는 설명

### 결정 D. 같은 기능군이라도 read 와 manage 를 나눠 적는다.

예시:
- 감사 로그 `열람` → `접근권한`
- 감사/보안 정책 `관리` → `관리자 권한`
- 조직도/직원 `조회` → `접근권한`
- 사용자 역할 `변경` → `관리자 권한`

즉 "같은 주제"라도
보는 권한과 바꾸는 권한을 다른 줄로 분리하는 것이 이번 카드의 핵심이다.

### 결정 E. 현재 4개 preview 토글은 아래처럼 재배치하는 방향을 권장한다.

현재 항목을 그대로 버리기보다 의미를 다시 배치한다.

권장 매핑:
- `조직 조회 모델` → `접근권한`
- `감사 로그 미리보기` → `접근권한`
- `사용자 역할 정책` → `관리자 권한`
- `기본 랜딩 보호` → `관리자 권한`

필요하면 builder 가 preview 항목을 1~2개 더 추가해
각 탭이 최소 2개 이상 항목을 자연스럽게 가지게 보강할 수 있다.
단, 새 항목도 모두 local state preview 범위 안에서만 추가한다.

### 결정 F. 탭마다 상태 요약 문구를 먼저 둔다.

권장 요약 문구:
- `접근권한`: "어디를 볼 수 있는지 정리합니다."
- `관리자 권한`: "무엇을 바꾸고 부여할 수 있는지 정리합니다."

이 요약은 짧고 단정해야 하며,
preview 단계라는 점을 숨기지 않아야 한다.

권장 보조 문구:
- "이번 미리보기에서는 선택 상태만 확인합니다. 실제 저장이나 운영 반영은 하지 않습니다."

### 결정 G. 고위험 느낌은 `관리자 권한` 쪽에서 더 분명히 준다.

`관리자 권한` 탭에는 아래 표현을 권장한다.
- 주의 필요
- 권한 변경 영향 있음
- 운영 반영은 별도 승인

반대로 `접근권한` 탭은
읽기/열람 중심이라 조금 더 차분한 설명을 쓴다.

## 5. 권장 UI 구조

### 5-1. 탭 구조
- 통합설정 상단 탭: `기본 설정` / `관리자설정`
- 관리자설정 내부 하위 탭: `접근권한` / `관리자 권한`

### 5-2. 각 하위 탭 공통 구성
1. 제목
2. 한 줄 설명
3. preview 안내 배너 또는 note
4. 항목 카드/토글 리스트
5. 선택 상태 요약 문장

### 5-3. 접근권한 탭 권장 예시 항목
- 관리자 화면 진입
- 조직/직원 조회
- 정책/운영 현황 열람
- 감사 로그 열람

### 5-4. 관리자 권한 탭 권장 예시 항목
- 사용자 역할 변경
- 사용자 상태 변경
- 기본 랜딩/접근 정책 관리
- 보안/감사 운영 관리

### 5-5. 표시 원칙
- label 만 봐도 read 와 manage 가 구분돼야 한다.
- 설명 문구는 "볼 수 있음"과 "바꿀 수 있음"을 섞지 않는다.
- 동일한 카드 스타일을 쓰되, `관리자 권한` 쪽은 경고 톤 note 를 둘 수 있다.

## 6. 포함 범위

### 이번 카드에 포함
- 관리자설정 내부 정보구조 재설계
- `접근권한` / `관리자 권한` 용어 고정
- preview/local state 기준 UI 항목/문구/상태 구조 정의
- 구현자가 바로 따라갈 acceptance 와 guardrail 정리
- 루트 문서(TASKS/HANDOFF/KNOWN_ISSUES/TEST_PLAN/CHANGELOG) 최소 동기화

### 다음 구현 카드에서 허용
- `mobile-app-shell.tsx` 관리자설정 UI 구조 변경
- local state preview 항목 재배치/추가
- 관련 웹 테스트 expectation 갱신
- 문구/레이블/상태 요약 반영

### 이번 카드에서 제외
- 실제 권한 저장
- 서버 API 연결
- DB schema/migration
- 감사 로그 적재/조회 backend 변경
- 운영 권한 실제 부여/회수
- secret 입력/교체
- production 데이터 변경

## 7. 구현 acceptance

1. 관리자설정에서 `접근권한` / `관리자 권한` 하위 탭이 분리되어 보인다.
2. 기존 상단 `기본 설정` / `관리자설정` 구조는 유지된다.
3. `접근권한` 탭 설명은 읽기/열람 중심으로만 적힌다.
4. `관리자 권한` 탭 설명은 변경/부여/관리 중심으로만 적힌다.
5. 같은 주제라도 read 와 manage 가 섞이지 않게 다른 항목으로 보인다.
6. 상태 선택/표시는 preview/local state 기준으로만 동작한다.
7. 실제 저장/API/DB/migration/audit log 반영이 없는 점을 숨기지 않는다.
8. 설정 진입 2차 비밀번호 게이트, 통합설정 첫 탭 `기본 설정`, `내정보 설정` 안 `2차 비밀번호 변경하기` 기준을 깨지 않는다.
9. 관리자설정 우상단 폐기 버튼은 다시 생기지 않는다.

## 8. 구현자에게 넘길 핵심 포인트

### gwbuilder
- `관리자설정` 안에 하위 탭을 추가하되 상단 탭 구조는 건드리지 않는다.
- 기존 4개 토글은 최대한 의미를 살려 두 탭으로 재배치한다.
- 각 탭 시작점에 짧은 설명과 preview note 를 둔다.
- read/write 의미가 섞이는 label 은 피한다.

### gwreviewer
- 탭 이름만이 아니라 설명 문장까지 read/manage 경계가 유지되는지 본다.
- 감사 로그, 조직 조회, 랜딩 정책 같은 항목이 잘못된 탭에 들어가지 않았는지 본다.
- 2차 비밀번호 관련 최신 UX 기준이 회귀하지 않는지 본다.

### gwtester
- 관리자설정 내부에서 하위 탭 전환이 정상인지 본다.
- 각 탭의 문구가 서로 다른 뜻으로 읽히는지 본다.
- 상단 `기본 설정` / `관리자설정`과 하위 `접근권한` / `관리자 권한`의 2단 구조가 테스트 기대값에 반영됐는지 본다.

## 9. 별도 승인 게이트

계속 별도 승인으로 남는 것:
- 실제 운영 권한 저장/회수
- 서버 권한 모델 확정
- DB migration
- 감사 로그 운영 반영
- secret 입력/교체
- production rollout
- 실데이터 변경

정리하면 이번 카드는
관리자설정 안에서 "볼 수 있는 권한"과 "바꿀 수 있는 권한"을
사용자가 바로 구분하게 만드는 preview UI 분리 작업 기준을 고정하는 단계입니다.
