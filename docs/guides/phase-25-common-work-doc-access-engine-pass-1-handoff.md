# Phase 25 공통 업무·문서·마감·권한 엔진 1차 handoff

한 줄 요약:
이번 Phase 25는
HR·세무·노무·법무·지점 운영 업무를 각각 따로 구현하기 전에,
모두가 같이 쓰는 공통 업무 카드, 문서, 첨부, 검토, 마감, 권한, 감사 로그 구조를
쉬운 말로 먼저 고정하는 단계입니다.

## 1. 지금 상태를 쉬운 말로 정리하면

이미 있는 기반:

- 직원 기본 업무 흐름은 이미 정리돼 있다.
- 관리자 운영 흐름과 파일럿 준비 흐름도 이미 정리돼 있다.
- 모바일 하단 탭 5개, `메뉴`/`홈`, PC sidebar 동일 정보구조 원칙도 있다.
- `지점/호텔 코드` 와 직원-지점 배정 초안도 문서에 올라와 있다.

아직 없는 것:

- HR·세무·노무·법무·지점 업무를 같이 담는 공통 work item 엔진
- 공통 검토/마감/문서/첨부 구조
- 같은 업무를 회사/지점/역할/capability 로 같이 묶는 통일된 설명

즉 지금은
"파일럿은 어떻게 돌릴지"
까지는 정리됐지만,
"그 다음 운영 업무를 어떤 공통 엔진으로 붙일지"
는 이번 Phase 25에서 처음 정리하는 셈입니다.

## 2. 이번 Phase를 어떻게 이해하면 되는가

### 1) 개별 모듈보다 공통 업무 카드를 먼저 만든다.

이번 단계에서 먼저 보는 것은
`인사 화면`, `세무 화면`, `법무 화면` 하나하나가 아니라
`공통 업무 항목(work item)` 입니다.

최소 공통 필드 예시:

- 어느 회사 업무인가
- 어느 지점/호텔 업무인가
- 어떤 모듈인가(`hr | tax | labor | legal | branch`)
- 제목/상태/우선순위는 무엇인가
- 누가 맡았는가
- 언제까지인가
- 검토가 필요한가
- 민감자료가 포함되는가

쉽게 말하면,
앞으로 업무 종류가 늘어도
"담는 그릇"
은 최대한 같게 하겠다는 뜻입니다.

### 2) 상태값도 공통 흐름으로 시작한다.

초기 공통 상태는 아래처럼 단순하게 봅니다.

- `draft`
- `todo`
- `in_progress`
- `waiting_review`
- `blocked`
- `done`
- `archived`

중요한 이유:

- 세무라서 특별 상태 10개,
- 법무라서 특별 상태 8개,
- 지점업무라서 또 다른 상태 6개
처럼 시작하면
처음부터 사용자와 운영자가 헷갈립니다.

그래서 1차는
상태를 통일하고,
모듈별 차이는 `module`, `category`, `review_required`, `contains_sensitive_data` 같은 필드로 푸는 방향입니다.

### 3) 문서/첨부/검토/마감은 업무 카드에 붙는 공통 구조로 본다.

이번 Phase 25에서 같이 묶는 것은 아래 5가지입니다.

1. 문서
   - 계약서, 증빙, 보고서, 요청서 같은 업무 관련 문서
2. 첨부
   - 파일 원문보다 metadata 중심으로 먼저 설명
3. 검토
   - 승인, 보완요청, 참고확인 같은 검토 기록
4. 마감
   - due date 와 마감 기준
5. 감사 로그
   - 누가 상태/담당자/권한/마감을 바꿨는지 기록

쉽게 말하면,
업무마다 문서 시스템을 따로 만들지 않고
같은 구조에 붙이는 방향입니다.

### 4) 권한은 회사 + 지점 + 역할 + capability 4축으로 본다.

이번 단계의 공통 원칙은 아래입니다.

- 회사가 다르면 기본 차단
- 같은 회사여도 지점/호텔 scope 를 다시 본다.
- 역할별로 기본 가시 범위를 나눈다.
- 상태 변경 가능 여부는 capability 로 한 번 더 본다.

예시:

- 일반 근무자: 자기 업무 또는 자기 지점 업무만 본다.
- 지점 관리자: 자기 지점 업무와 지점 운영 검토를 본다.
- 본사 관리자: 여러 지점을 묶어 본다.
- HR/세무/노무/법무 담당자: 승인된 모듈 범위만 본다.
- 감사 사용자: read-only 로 추적 중심 확인을 한다.

중요:
단순히 메뉴를 숨길지만 정하는 것이 아니라,
보이는 범위와 바꿀 수 있는 범위를 같이 설계해야 합니다.

### 5) 모바일 하단 탭은 늘리지 않는다.

Phase 25에서도 아래는 그대로 유지합니다.

- 하단 탭: `메뉴`·`홈`·`메신저`·`메일`·`알림`

새 업무 모듈 자리는 아래처럼 풉니다.

- 모바일 `홈`: 내 업무, 마감 임박, 검토 대기 요약 카드
- 모바일 `메뉴`: `인사`, `세무`, `노무`, `법무`, `지점 업무` 또는 공통 `업무` 그룹
- PC sidebar: 일반 업무 그룹과 관리자 운영 그룹 분리

즉,
새 업무가 생긴다고 하단 탭부터 늘리는 것이 아니라
기존 정보구조 안에서 자리를 먼저 확보합니다.

### 6) 민감 자료와 실처리는 계속 승인 게이트다.

이번 단계에서도 아직 열지 않는 것:

- production DB 실데이터
- 주민번호/계좌/급여/신고자료 같은 민감 원문
- 실제 세무/노무/법무 실처리
- 외부 세무사/노무사/변호사 계정 연동
- 홈택스/전자서명/메일 실연동
- 자동 신고/자동 제출/자동 계산 결과 확정

이 문서는
"나중에 이런 업무를 담을 그릇"
을 정하는 단계지,
실제 민감 업무를 바로 시작하는 단계가 아닙니다.

## 3. 대장이 가장 먼저 볼 8가지 질문

1. 앞으로 HR·세무·노무·법무·지점 업무를 어떤 공통 업무 카드 구조로 담을 것인가?
2. 상태값을 모듈마다 따로 늘리지 않고 어떤 공통 상태 흐름으로 시작하는가?
3. 문서, 첨부, 검토, 마감, 감사 로그를 업무 카드에 어떤 방식으로 붙이는가?
4. 회사 + 지점/호텔 + 역할 + capability 접근 기준이 같은 언어로 설명되는가?
5. 일반 근무자 / 지점 관리자 / 본사 관리자 / 감사 사용자의 가시 범위가 구분되는가?
6. 모바일 하단 탭을 늘리지 않고도 새 업무 모듈 자리를 `홈`/`메뉴`/PC sidebar 에 확보하는가?
7. builder가 먼저 구현할 최소 안전 범위가 공통 engine Production-ready (실구현) 중심으로 정리돼 있는가?
8. 민감 문서, 실데이터, 외부 연동, 실제 신고/제출 자동화가 여전히 승인 게이트로 남아 있는가?

이 8개 질문에 바로 답이 안 보이면
이번 Phase 25 정리가 덜 된 상태입니다.

## 5. 권장 구현 순서

1. 공통 work item 필드와 상태값을 먼저 맞춘다.
2. 문서/첨부/검토/마감/감사 로그를 work item에 붙이는 구조를 정리한다.
3. 회사 + 지점 + 역할 + capability 접근 기준을 Production-ready (실구현) copy 와 API 설명에 같이 반영한다.
4. 모바일 `홈`/`메뉴`, PC sidebar 에 새 업무 그룹 자리를 과장 없이 확보한다.
5. `/work-items`, `/hr`, `/tax`, `/labor`, `/legal`, `/branch/tasks`, `/admin/work-items` 같은 route 초안 중 최소 Production-ready (실구현) 범위를 정한다.
6. 실민감 자료/외부 연동/실처리 항목이 문서 중간에서 구현 TODO처럼 섞이지 않았는지 마지막에 다시 본다.

## 6. 각 역할 카드에 넘길 핵심 포인트

### 구현자(gwbuilder)

집중할 것:

- 공통 work item contract
- 공통 문서/첨부/검토/마감 schema 초안
- Production-ready (실구현) route 또는 응답 Production-ready (실구현)
- `홈`/`메뉴`/PC sidebar 새 업무 그룹 자리 확보
- 지점 scope 와 역할 scope 를 함께 설명하는 copy

피해야 할 것:

- 실제 급여/세무/노무 계산 엔진
- 민감 문서 원문 저장
- production DB 실데이터 연결
- 외부 전문가/정부/메일/전자서명 시스템 연동

### 리뷰어(gwreviewer)

집중할 것:

- 공통 엔진 설명이 개별 모듈 완성처럼 과장되지 않는지
- 상태값과 권한 규칙이 모듈마다 갈라지지 않는지
- 민감 자료와 실처리 항목이 승인 게이트로 남아 있는지

### 테스터(gwtester)

집중할 것:

- 공통 contract, route, Production-ready (실구현), 메뉴 자리가 문서와 맞는지
- 일반 근무자/지점 관리자/본사 관리자 가시 범위가 섞이지 않는지
- 막힘이 권한/자료부족/승인필요/Production-ready (실구현) 제한 중 무엇인지 나눠 기록되는지

### 문서화(gwdocs)

집중할 것:

- 업무 카드, 문서, 검토, 마감, 감사 로그 언어를 쉬운 한국어로 통일
- 모듈별 차이보다 공통 엔진 구조를 먼저 설명
- 민감자료/실처리/승인게이트 경계 반복 고정

## 7. builder가 바로 옮겨 적을 최소 구현 메모

- shared contract 이름은 work item 중심으로 단순하게 시작한다.
- 모듈 enum 은 `hr | tax | labor | legal | branch` 로 먼저 고정한다.
- status enum 은 공통 상태 7개로 먼저 시작한다.
- 문서/첨부 구조는 metadata 중심으로 두고 raw storage 정보는 비노출 원칙을 유지한다.
- `review_required`, `contains_sensitive_data`, `branch_id?`, `assignee_role_code?` 같은 필드로 후속 확장을 대비한다.
- 일반 업무 메뉴와 관리자 운영 메뉴를 섞지 않는다.
- 지점 미배정 사용자는 `지점 배정 필요` 또는 scope 부족 안내를 먼저 보게 한다.

## 8. 이번 Phase에서 하지 않는 것

- 실제 급여 계산 확정
- 실제 세무 신고/제출 자동화
- 실제 노무 사건 처리 확정
- 실제 법률 자문 시스템 연동
- 외부 전자서명/메일/메신저/정부 시스템 연동
- production migration
- 실데이터 기반 branch master 확정

## 9. 마지막 체크

완료 전에 아래를 다시 봅니다.

- 문서가 공통 엔진을 설명하는지, 개별 모듈 완성을 과장하지 않는지
- 지점/역할/권한 경계가 루트 문서와 같은 뜻인지
- 모바일 하단 탭을 늘리지 않는 원칙이 깨지지 않았는지
- 민감 문서/실데이터/외부 연동이 계속 승인 게이트인지
- 다음 builder/reviewer/tester/docs 카드가 바로 이어갈 수 있는지
