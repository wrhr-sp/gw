# 관리자설정 접근권한/관리자 권한 탭 분리 handoff

한 줄 요약:
이번 1차는 통합설정 안의 `관리자설정` 화면에서
"어디를 볼 수 있는가"와 "무엇을 바꿀 수 있는가"를 섞지 않도록
`접근권한` / `관리자 권한` 하위 탭으로 다시 나누는 preview UI 기준 정리입니다.

## 1. 지금 상태를 쉬운 말로 정리하면

이미 있는 것:
- 설정 관련 진입 전에 2차 비밀번호 4칸 PIN 게이트가 있다.
- 통합설정 상단 탭은 `기본 설정` / `관리자설정` 두 개다.
- `관리자설정` 안에는 현재 `관리자 권한 확인` 카드와 토글 4개가 있다.
- 현재 모든 상태는 local state preview 다.

지금 부족한 것:
- "읽기/열람 권한"과 "변경/부여 권한"이 한 카드에 섞여 있다.
- 감사 로그 열람, 조직 조회, 역할 정책, 랜딩 보호가 같은 종류의 권한처럼 보인다.
- 후속 구현자가 read capability 와 manage capability 를 나눠 읽기 어렵다.

즉 지금은
관리자설정이 아예 없는 상태가 아니라,
의미 분리가 약한 상태입니다.

## 2. 이번 카드에서 고정한 핵심 해석

### 1) 상단 탭 구조는 유지한다.
그대로 유지:
- `기본 설정`
- `관리자설정`

새로 나누는 곳:
- `관리자설정` 내부
  - `접근권한`
  - `관리자 권한`

### 2) `접근권한`은 "볼 수 있음" 언어만 쓴다.
예:
- 관리자 화면 진입
- 조직/직원 조회
- 정책/운영 현황 열람
- 감사 로그 열람

쓰면 좋은 말:
- 조회
- 열람
- 들어가기
- 볼 수 있음

피해야 할 말:
- 변경
- 부여
- 관리
- 실행

### 3) `관리자 권한`은 "바꿀 수 있음" 언어만 쓴다.
예:
- 사용자 역할 변경
- 사용자 상태 변경
- 기본 랜딩/접근 정책 관리
- 보안/감사 운영 관리

쓰면 좋은 말:
- 변경
- 부여
- 관리
- 편집
- 실행

피해야 할 말:
- 단순 조회/열람 중심 설명

### 4) 같은 주제라도 read 와 manage 를 나눠 적는다.
예:
- 감사 로그 열람 → `접근권한`
- 감사/보안 정책 관리 → `관리자 권한`
- 조직/직원 조회 → `접근권한`
- 사용자 역할 변경 → `관리자 권한`

이 분리가 이번 카드의 핵심입니다.

## 4. builder가 바로 적용할 권장 방향

### A. 관리자설정 내부에 하위 탭 state를 추가한다.
추천 형태:
- `access`
- `adminPermissions`

중요:
- 기존 `activeSettingsTab`(`basic` / `admin`) 구조는 유지
- 새 state 는 `관리자설정` 내부에서만 쓴다

### B. 기존 4개 preview 토글은 우선 아래처럼 재배치한다.
권장 매핑:
- `조직 조회 모델` → `접근권한`
- `감사 로그 미리보기` → `접근권한`
- `사용자 역할 정책` → `관리자 권한`
- `기본 랜딩 보호` → `관리자 권한`

필요하면 preview 항목 1~2개를 더 보강해도 된다.
단, local state 범위를 넘지 않는다.

### C. 각 탭 시작점에 짧은 설명을 둔다.
권장 문구 예시:
- `접근권한`: "어디를 볼 수 있는지 정리합니다."
- `관리자 권한`: "무엇을 바꾸고 부여할 수 있는지 정리합니다."
- 공통 note: "이번 미리보기에서는 선택 상태만 확인합니다. 실제 저장이나 운영 반영은 하지 않습니다."

### D. `관리자 권한` 쪽에는 주의 톤을 조금 더 준다.
예:
- `주의 필요`
- `권한 변경 영향 있음`
- `운영 반영은 별도 승인`

반면 `접근권한`은 읽기/열람 중심으로 차분하게 둔다.

## 5. 절대 깨지면 안 되는 기준

1. 설정 진입 전 2차 비밀번호 게이트 유지
2. 통합설정 첫 진입 탭은 계속 `기본 설정`
3. 관리자설정 우상단 폐기된 `2차 비밀번호 변경` 버튼 부활 금지
4. `내정보 설정` 안 `2차 비밀번호 변경하기` 버튼 유지
5. 이번 단계는 저장/API/DB/migration/audit log 운영 반영 없음

즉 탭을 나누더라도
보안 게이트와 최신 UX 합의는 그대로 유지해야 합니다.

## 6. reviewer가 볼 핵심 질문

1. `접근권한` 탭 설명에 변경/부여/관리 같은 말이 섞이지 않았는가?
2. `관리자 권한` 탭 설명에 단순 조회/열람 중심 문장이 섞이지 않았는가?
3. 감사 관련 항목이 열람과 관리로 분리돼 있는가?
4. 조직 조회와 사용자 역할 변경이 같은 그룹처럼 보이지 않는가?
5. 상단 탭 구조와 2차 비밀번호 최신 기준이 회귀하지 않았는가?

## 7. tester가 볼 핵심 질문

1. 관리자설정에 하위 탭 `접근권한` / `관리자 권한` 이 보이는가?
2. 탭 전환 시 각 항목군이 분리되어 보이는가?
3. 상단 `기본 설정` / `관리자설정` 흐름은 그대로인가?
4. preview/local state 토글이 각 탭에서 정상 표시되는가?
5. read/manage 문구가 서로 다른 의미로 읽히는가?
6. 2차 비밀번호 관련 기존 기대값이 깨지지 않았는가?

## 8. 권장 acceptance

- `관리자설정` 내부에 `접근권한` / `관리자 권한` 하위 탭이 생긴다.
- `기본 설정` / `관리자설정` 상단 2탭 구조는 그대로다.
- `접근권한`은 열람/조회/진입 의미만 보여 준다.
- `관리자 권한`은 변경/부여/관리 의미만 보여 준다.
- 현재 preview/local state 범위를 넘어가지 않는다.
- 실제 저장, API, DB, migration, production 반영이 없는 점을 숨기지 않는다.

## 9. 이번 카드에서 하지 않는 것

- 실제 관리자 권한 저장/회수
- 서버 권한 모델 확정
- DB schema/migration
- backend 감사 로그 처리 변경
- secret 입력/교체
- production rollout

정리하면 다음 작업자는
`관리자설정`을 통째로 새로 만드는 것이 아니라,
기존 preview UI 안에서 권한 의미를 두 갈래로 더 선명하게 나누면 됩니다.
