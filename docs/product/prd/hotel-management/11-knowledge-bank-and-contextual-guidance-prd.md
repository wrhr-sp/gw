# 호텔 운영 지식뱅크·기능 가이드 PRD

## 1. 문서 정보

| 버전 | `1.0-user-approved` |

## 2. 목표

정식 운영 매뉴얼로 일반화하기 어렵지만 호텔 운영에 필요한 상황별 대응법·경험·실패사례를 검토 가능한 지식으로 축적하고, 직원이 현장에서 빠르게 검색·재사용하게 한다. 동시에 모든 사용자 기능의 제목 옆에 `?` 가이드를 제공해 기능의 목적·사용대상·핵심 사용법·주의사항을 화면을 벗어나지 않고 확인하게 한다.

## 3. 책임 경계

### 3.1 정식 운영 매뉴얼

- 반드시 준수해야 하는 승인된 정책·표준절차·역할·체크리스트다.
- 안전·보안·개인정보·법적 의무·보상 승인한도와 같이 임의 판단할 수 없는 기준을 포함한다.
- 강한 승인·버전·변경통제를 적용한다.

### 3.2 지식뱅크

- 모든 호텔과 상황에 일률적으로 적용하기 어려운 실무 노하우·진단법·사례·교훈을 제공한다.
- 참고자료이며 게시만으로 의무절차나 승인된 정책이 되지 않는다.
- 매뉴얼·안전규정·법적 의무·관리자 지시와 충돌하면 공식 기준을 우선한다.
- 검증된 지식은 별도 승인·버전 절차를 거쳐 매뉴얼 후보로 제안할 수 있으나 자동 승격하지 않는다.

### 3.3 실제 업무 레코드

- 하자·운영이슈·컴플레인·작업지시의 상태, 담당자, 고객과의 실제 대화, 증거, 조치 이력은 해당 업무 레코드가 정본이다.
- 지식 글은 실제 사건을 대체하거나 사건을 완료 처리하지 않고 해결 참고자료로 연결된다.
- 사건 해결 후 새 교훈은 기존 지식의 수정 제안 또는 새 초안으로 남긴다.

### 3.4 기능 가이드

- 제품 기능의 간단한 사용 설명이며 운영 지식 글이나 매뉴얼을 대체하지 않는다.
- 기능별 고정 식별자와 제품 버전에 맞춰 관리하는 제품 콘텐츠다.
- 권한 없는 기능·자료의 존재, 내부 식별자, 타 호텔 정보는 가이드로 노출하지 않는다.

## 4. 지식뱅크 초기 범위

### 포함

1. 지식 목록·상세·검색·필터
2. 초안 작성·수정·검토 요청·게시·재검토·보관
3. 회사 공통 또는 특정 호텔 적용범위
4. 하자보수·시설설비·컴플레인 대응·객실운영·하우스키핑·안전주의·협력업체·기타 분류
5. 증상·상황, 확인순서, 권장대응, 금지·주의사항, 에스컬레이션 기준, 사례·결과·교훈
6. 태그, 관련 매뉴얼, 운영이슈·컴플레인·작업지시 링크
7. 비공개 첨부자료와 작성·검토·게시·보관 감사
8. 최종 검토일·재검토 예정일과 오래된 지식 표시
9. 모바일 현장 검색·열람
10. 유용성 평가와 오류·수정 신고

### 제외

- 외부 고객 공개 지식베이스
- 자유 게시판·실시간 채팅·댓글 커뮤니티
- AI 생성답변 또는 AI가 검토 없이 자동 게시하는 지식
- PMS·OTA·전화·메신저의 컴플레인 자동수집
- 지식만으로 환불·보상·객실 판매중지·안전조치를 자동 승인하는 기능
- 승인 절차 없는 매뉴얼 자동 승격

## 5. 지식 유형과 예시

| 유형 | 예시 |
|---|---|
| 하자보수 | 에어컨 냉방 저하 점검순서, 누수 위치별 원인 후보, 도어락 반복오류 확인법 |
| 시설·설비 | 직원이 가능한 안전한 임시조치, 객실 판매중지 또는 전문업체 호출 기준 |
| 컴플레인 대응 | 소음·청결·냄새 민원의 사실확인, 권장 표현, 피해야 할 표현, 관리자 보고 기준 |
| 객실·하우스키핑 | 반복되는 객실 특이사항, 청소·점검 중 발견한 예외 대응 |
| 협력업체 | 설비·지역업체 호출 전 확인사항과 처리 경험. 계약상 비밀·개인 연락처는 제한 |
| 성공·실패 사례 | 효과가 있었던 대응, 악화시킨 대응, 재발 방지 교훈 |

## 6. 지식 데이터 구조

필수 또는 조건부 필드는 다음과 같다.

```text
id, company_id
scope_type(COMPANY | HOTEL), hotel_id(nullable)
title, summary, knowledge_type
situation, symptoms_and_context, checks
recommended_response, prohibited_or_caution_response
escalation_criteria, required_permission_or_approval
case_summary, outcome_and_lesson
tags, related_manual_refs
related_issue_ids, related_complaint_ids, related_work_order_ids
status, author_user_id, reviewer_user_id
published_at, reviewed_at, review_due_at
version, created_at, updated_at
```

- 회사 공통이면 `hotel_id`는 비어 있고, 호텔 전용이면 같은 회사의 유효 호텔을 반드시 참조한다.
- 실제 사건 링크는 동일 회사·허용 호텔·자료권한을 모두 통과한 사용자에게만 표시한다.
- 관련 사건을 삭제하거나 상태를 변경하지 않고 참조만 저장한다.
- 첨부는 비공개 R2와 공통 파일 권한·검역·감사를 사용한다.

## 7. 생애주기

```text
DRAFT
→ REVIEW_REQUESTED
→ PUBLISHED
→ NEEDS_REVIEW
→ PUBLISHED 또는 ARCHIVED
```

- 작성 권한과 게시 권한을 분리한다.
- 초안·검토요청은 일반 검색결과에 노출하지 않는다.
- 게시 이후 본문 변경은 새 version이며 작성자·검토자·변경사유를 보존한다.
- 재검토 예정일이 지나면 `NEEDS_REVIEW`로 표시하고 검색결과에서 오래된 정보임을 명확히 알린다.
- 안전·법률·개인정보·환불·보상 관련 내용은 지정 검토자만 게시할 수 있다.
- 보관된 글은 신규 업무에서 기본 추천하지 않지만 과거 사건 재현을 위해 물리삭제하지 않는다.

## 8. 권한

초기 권한코드는 다음처럼 분리한다.

```text
KNOWLEDGE_READ
KNOWLEDGE_CREATE
KNOWLEDGE_REVIEW
KNOWLEDGE_PUBLISH
KNOWLEDGE_ARCHIVE
```

- 모든 판정은 `company_id ∩ 호텔범위 ∩ 사용자유형 ∩ 기능권한 ∩ 자료상태`를 서버에서 확인한다.
- `KNOWLEDGE_CREATE`는 본인 초안 작성 권한일 뿐 게시 권한이 아니다.
- 게시·보관은 별도 권한과 사유를 요구한다.
- 하우스키핑과 호텔 소유주는 명시적으로 허용된 회사 공통 또는 자기 호텔 게시자료만 읽는다.
- 타 호텔 전용 지식의 제목·건수·태그·검색어 자동완성도 노출하지 않는다.

## 9. 검색과 현장 사용

- 제목뿐 아니라 요약, 상황, 증상, 대응, 시설·객실·업무영역, 태그를 검색한다.
- 필터는 적용범위, 호텔, 유형, 상태, 최종 검토일을 제공한다.
- 검색은 회사·호텔 권한 필터를 먼저 적용한 뒤 관련도를 계산한다.
- 결과에는 적용범위, 게시·재검토 상태, 최종 검토일과 핵심 대응요약을 표시한다.
- 모바일 상세 상단에는 `먼저 확인할 사항 → 가능한 대응 → 하지 말아야 할 대응 → 에스컬레이션` 순서로 배치한다.
- 오래된 지식·보관 지식·검토되지 않은 초안을 최신 권장답변처럼 표시하지 않는다.

## 10. 개인정보·안전·법률 통제

지식 글에는 다음 원문을 저장하지 않는다.

- 고객 실명, 전화번호, 이메일, 예약번호, 결제·신분증 정보
- 얼굴이 식별되는 사진, 대화 전문, 건강정보 등 민감정보
- 비밀번호, token, 출입코드, 비공개 설비 보안정보
- 계약상 비밀인 업체정보나 승인되지 않은 개인 연락처

사례는 개인을 식별할 수 없게 일반화한다. 개인정보 가능성이 있는 입력·첨부는 게시 전 경고·검토하며, 신고 접수 시 즉시 비공개 전환할 수 있다. 생명·화재·가스·전기·범죄·법률·보상승인 상황은 지식만 보고 종결하지 않고 공식 비상절차와 승인자를 안내한다.

## 11. 기능 제목 옆 공통 가이드

### 11.1 적용범위

- 호텔관리의 모든 사용자-facing 기능 페이지의 `PageHeader` 제목 옆에 원형 `?` 가이드 버튼을 둔다.
- 목록·상세·작성·설정처럼 독립된 주요 기능에는 각각 해당 화면 설명을 제공한다.
- 모든 작은 카드·필드·표 열에 반복 표시해 화면을 혼잡하게 만들지 않는다. 필드 설명이 필요하면 label 도움말 패턴을 별도로 사용한다.
- 사용자가 볼 권한이 없는 기능의 가이드 버튼과 콘텐츠도 렌더하지 않는다.

### 11.2 가이드 내용

가이드는 짧고 쉬운 한국어로 다음 항목을 최대한 간결하게 제공한다.

```text
이 기능은 무엇을 하는가
누가 사용할 수 있는가
기본 사용순서
필요한 권한·승인
중요한 주의사항
필요 시 관련 매뉴얼·지식뱅크·상세 도움말 링크
```

- 현재 화면과 일치하지 않는 예정 기능·가짜 예시·내부 기술용어를 표시하지 않는다.
- 위험 작업은 가이드 확인만으로 승인된 것으로 취급하지 않는다.
- 가이드는 사용자의 업무입력, 비밀번호, 고객정보를 수집하지 않는다.

### 11.3 상호작용·접근성

- `?`는 장식 아이콘이 아니라 실제 `button`이며 접근 가능한 이름을 `"{기능명} 도움말"`로 제공한다.
- 클릭·터치·Enter·Space로 열고 `Escape`, 닫기 버튼, 바깥영역으로 닫을 수 있다.
- 열린 콘텐츠는 PC에서 Popover, 좁은 모바일에서 Sheet 또는 Dialog로 표시할 수 있지만 내용과 의미는 동일하다.
- 열릴 때 포커스를 안전하게 이동하고 닫을 때 원래 `?` 버튼으로 복귀한다.
- 모바일 터치영역은 최소 44×44px, WCAG AA 대비를 충족하며 색상·hover에만 의존하지 않는다.
- 툴팁 hover만으로 구현하지 않는다. 키보드·터치·스크린리더에서 동일 정보에 접근해야 한다.
- 한 화면에서 하나의 가이드만 열고 본문 스크롤·뒤로가기·저장 중 상태를 방해하지 않는다.

### 11.4 콘텐츠 관리

- 각 가이드는 안정적인 `feature_key`, 제목, 요약, 대상 사용자, 사용순서, 권한·주의사항, 관련 링크, 콘텐츠 버전을 가진다.
- 초기 MVP는 코드와 함께 검토·배포되는 typed feature-guide registry를 정본으로 사용한다. 운영자가 임의 HTML·script를 입력하는 CMS는 제외한다.
- 기능 변경 PR에는 관련 가이드 갱신 또는 `변경 없음` 근거가 반드시 포함된다.
- 없는 `feature_key`, 빈 설명, 현재 route와 불일치한 가이드는 build/test에서 실패시킨다.
- 가이드 열람은 일반적으로 감사대상이 아니며, 개인별 열람이력·프로파일링을 저장하지 않는다.

### 11.5 현재 구현 범위

- 계정관리 1차 slice는 사용자 목록, 사용자 생성, 사용자 상세 route에 각각 `account-administration.list`, `account-administration.create`, `account-administration.detail` 가이드를 연결한다.
- 공통 `FeatureGuide`는 PC에서 Radix Popover, 390px 모바일에서 Radix Dialog를 사용하며 같은 registry 콘텐츠를 표시한다.
- 계정관리 route coverage, registry 필수 콘텐츠, 내부용 문구 금지, 44×44px trigger, Escape·바깥영역 닫기와 trigger 포커스 복귀를 자동 검증한다.
- 이 1차 slice 완료를 호텔관리 모든 사용자-facing route의 guide coverage 완료로 간주하지 않는다. 다른 기능은 각 실제 UI·권한 계약 구현과 함께 같은 공통 패턴으로 추가한다.

## 12. 제안 API 경계

```text
GET  /api/knowledge
POST /api/knowledge
GET  /api/knowledge/:knowledgeId
PATCH /api/knowledge/:knowledgeId
POST /api/knowledge/:knowledgeId/transitions
POST /api/knowledge/:knowledgeId/feedback
GET  /api/knowledge/:knowledgeId/links
```

기능 가이드는 초기 MVP에서 정적 typed registry를 사용하므로 별도 변경 API를 두지 않는다. 실제 route·Zod 계약은 구현 전에 `packages/contracts`에 먼저 확정한다.

## 13. 감사·멱등·동시성

- 지식 생성·수정·검토요청·게시·재검토·보관·오류신고 처리에 행위자, 회사, 호텔, 전후 최소요약, 사유, 결과, trace ID를 남긴다.
- 변경 API는 멱등키와 `version` 낙관적 잠금을 적용한다.
- 게시 전 검토대상 version과 실제 게시 version이 다르면 `409`로 차단한다.
- 감사요약에 고객정보·첨부 원문·검색어 원문을 저장하지 않는다.

## 14. 수용 기준

### 지식뱅크

- 회사 공통 게시 지식과 자기 호텔 게시 지식만 검색·열람된다.
- 타 회사·타 호텔 지식은 제목·건수·태그·링크 존재도 노출되지 않는다.
- 초안 작성자가 게시권한 없이 직접 `PUBLISHED`로 전환할 수 없다.
- 관련 하자·컴플레인 링크는 원자료 권한이 없으면 필드 자체를 비노출한다.
- 개인정보가 포함된 사례는 게시되지 않고 안전한 검토오류를 반환한다.
- 재검토 기한이 지난 글은 최신 권장지식처럼 표시되지 않는다.
- 모바일에서 증상 검색 후 핵심 확인·대응·금지·에스컬레이션을 주요 행동 전에 확인할 수 있다.
- 게시된 글의 과거 version·검토자·변경사유를 재현할 수 있다.

### 기능 가이드

- 모든 사용자-facing 기능 페이지의 제목 옆에 해당 기능의 `?` 버튼이 있다.
- 버튼을 누르면 목적·대상·기본 사용법·권한·주의사항이 짧은 한국어로 표시된다.
- 키보드와 스크린리더로 열기·읽기·닫기·포커스 복귀가 가능하다.
- 390px 모바일에서 최소 44px 터치영역과 Sheet/Dialog 읽기 흐름을 충족한다.
- 권한 없는 기능의 제목·가이드·관련 링크가 노출되지 않는다.
- 기능 route와 가이드 registry 누락·중복·빈 콘텐츠를 자동 테스트가 차단한다.
- 가이드가 실제 API 권한검사·확인 Dialog·재인증·승인절차를 우회하지 않는다.
