# 계정관리 정책 결정 원장

## 문서 정보

- 대상 PRD: `10-account-administration-prd.md`
- 상태: 초기 MVP 정책 확정 · 후속 확장 질문 보류
- 최종 갱신: 2026-07-22
- 적용 범위: 호텔관리 로그인 ID와 ZITADEL 로그인명 연결 정책

## 적용 단계

### 초기 MVP 구현 대상

- `LOGIN-ID-001`: 위아히어 전체 canonical 로그인 ID 유일성
- `LOGIN-ID-002`: ASCII 영문 소문자·숫자 3~30자
- `LOGIN-ID-003`: 계정 생성 관리자가 짧은 ID 직접 입력
- `LOGIN-ID-004`: 짧은 ID의 검증된 ZITADEL subject 서버 해석·provider 전체 로그인명 사용자 비노출
- `LOGIN-ID-005`: 과거 사용자 ID를 다른 사람에게 재사용 금지. 동일인 재활성화 API 세부는 후속 구현
- `LOGIN-ID-009` 중 필수 고정 예약 ID 차단

### 후속 확장 구현 대상

- `LOGIN-ID-006~008`: 관리자 로그인 ID 변경, 전용 capability, TOTP 재인증
- `LOGIN-ID-009` 중 회사별 추가 예약 목록
- `LOGIN-ID-010~011`: 추가 예약 ID 전용 관리권한과 조건부 2단계 해제
- 회사별 추가 예약 ID의 전역 ID 공간 충돌 처리

후속 확장 정책은 사용자 확정 결정을 보존하되 초기 MVP 구현·release gate에 포함하지 않는다. 초기 MVP 완료 전에는 후속 확장 질문을 계속하지 않는다.

## 구현 정렬 상태

- 전역 `login_id_registry`가 provider 호출 전에 canonical ID를 선점하며 UPDATE·DELETE 금지 trigger로 과거 ID tombstone을 영구 보존한다.
- `users`는 전역 unique index와 registry 복합 FK를 함께 사용해 claim 없는 로그인 ID 저장을 차단한다.
- 로그인 BFF는 짧은 ID를 canonicalize한 뒤 PostgreSQL의 활성 identity mapping에서 검증된 ZITADEL subject를 해석하고 Session API에 `userId`로 전달한다.
- Preview bootstrap은 provider username을 자동 변경하지 않고 기존 검증된 subject를 유지한 채 호텔관리 alias를 `previewadmin`으로 정렬한다.
- 실제 Preview 신규 ID 로그인·이전 alias 거부·subject/권한 불변 smoke 전에는 rollout 완료로 보지 않는다.

## 확정 결정

### LOGIN-ID-001 로그인 ID 유일성

- 상태: `confirmed`
- 질문: 로그인 ID의 중복 허용 범위
- 사용자 답변: 위아히어 전체에서 ID를 유일하게 유지한다.
- 확정 규칙: 하나의 canonical 로그인 ID는 회사와 관계없이 위아히어 전체 계정 영역에서 유일해야 한다. 회사별로 동일한 로그인 ID를 중복 발급하지 않는다. 비활성·과거 사용자 ID의 재사용 금지는 `LOGIN-ID-005`를 따른다.
- 영향:
  - 로그인할 때 회사코드 입력이나 회사 선택을 요구하지 않는다.
  - 로그인 ID 중복 검사는 대소문자를 정규화한 값으로 전체 범위에서 수행한다.
  - 중지된 ID는 `LOGIN-ID-005`에 따라 다른 사용자에게 재사용하지 않는다.

### LOGIN-ID-002 로그인 ID 문자와 길이

- 상태: `confirmed`
- 질문: 로그인 ID에 허용할 문자와 형식
- 사용자 답변: 영문 소문자·숫자만, 3~30자.
- 확정 규칙:
  - canonical 로그인 ID는 ASCII 영문 소문자 `a-z`와 숫자 `0-9`만 허용한다.
  - 길이는 Unicode 코드 포인트 기준 3~30자로 제한한다.
  - 사용자가 영문 대문자를 입력하면 서버와 UI에서 소문자로 정규화한다.
  - 공백, 한글, 이메일 주소 구분자 `@`, 점, 밑줄, 하이픈은 허용하지 않는다.
- 영향:
  - API·DB·ZITADEL provisioning 전에 같은 정규화·검증 규칙을 적용해야 한다.
  - 이메일과 사원번호는 로그인 ID와 별도 속성으로 유지한다.

### LOGIN-ID-003 계정 생성 시 ID 결정 주체

- 상태: `confirmed`
- 질문: 계정 생성 시 로그인 ID 결정 방식
- 사용자 답변: 관리자가 로그인 ID를 직접 입력한다.
- 확정 규칙: 계정 생성 권한이 있는 관리자가 호텔관리 계정 생성 화면에서 로그인 ID를 직접 입력하며, 사원번호나 이메일 앞부분으로 자동 확정하지 않는다.
- 영향:
  - 저장 전 형식·전체 유일성 검사를 수행한다.
  - 사원번호 또는 이메일 기반 자동 제안 여부는 현재 확정 범위에 포함하지 않는다.

### LOGIN-ID-004 ZITADEL 계정주소 노출과 로그인 입력

- 상태: `confirmed`
- 질문: ZITADEL의 `ID@조직주소`를 사용자에게 어떻게 처리할 것인가
- 사용자 답변: 짧은 ID만 입력하고 ZITADEL 전체 주소는 서버가 자동 결합·숨긴다.
- 확정 규칙:
  - 계정 생성과 로그인 UI는 짧은 로그인 ID만 입력받는다.
  - 서버는 승인된 환경별 ZITADEL organization과 PostgreSQL identity mapping으로 짧은 ID에 대응하는 검증된 provider subject를 해석한다.
  - ZITADEL Session API에는 검증된 `userId`를 전달하며, 요청에서 organization suffix나 전체 ZITADEL 로그인명을 받거나 신뢰하지 않는다.
  - 전체 ZITADEL 로그인명은 계정 생성·상세·목록·로그인 오류·감사 화면에 노출하지 않는다.
  - 일반 로그인에서 전체 ZITADEL 로그인명을 호환 입력으로 허용하지 않는다.
- 영향:
  - ZITADEL Cloud에서 자체 호스팅 ZITADEL로 이전해도 사용자 로그인 ID는 유지한다.
  - provider identity 연결은 전체 로그인명 대신 검증된 ZITADEL subject와 호텔관리 내부 사용자 ID를 정본으로 사용한다.

### LOGIN-ID-005 비활성·중지 계정의 로그인 ID 재사용

- 상태: `confirmed`
- 질문: 중지된 계정의 로그인 ID를 다른 사용자가 다시 사용할 수 있는가
- 사용자 답변: 다른 사람에게 영구 재사용을 금지하고, 동일인 복귀는 기존 계정을 재활성화한다.
- 확정 규칙:
  - 한 번 발급한 canonical 로그인 ID는 계정이 비활성·중지돼도 다른 사용자에게 다시 발급하지 않는다.
  - 동일인이 복귀하면 새 사용자나 새 로그인 ID를 만들지 않고, 본인·인사 식별자·기존 provider identity를 검증한 뒤 기존 계정을 재활성화한다.
  - 이름이나 이메일 일치만으로 동일인으로 판정하지 않는다.
  - 재활성화 전 현재 회사·사용자유형·호텔배정·역할·권한을 새로 검증하며 과거 권한을 자동 복원하지 않는다.
- 영향:
  - 중복 검사는 활성 계정뿐 아니라 보존된 비활성 계정의 canonical 로그인 ID까지 포함한다.
  - 동일인 재활성화 API·승인권자·감사사건은 후속 구현 범위에서 별도로 명세한다.

### LOGIN-ID-006 로그인 ID 변경

- 상태: `confirmed`
- 질문: 계정 생성 이후 로그인 ID를 변경할 수 있는가
- 사용자 답변: 권한 있는 관리자만 예외적으로 변경하고 모든 로그인 세션을 회수한다.
- 확정 규칙:
  - 일반 사용자는 자신의 로그인 ID를 변경할 수 없다.
  - 별도 권한이 확인된 관리자만 오타 교정이나 승인된 계정체계 변경 사유로 로그인 ID를 변경할 수 있다.
  - 새 ID는 동일한 전체 유일성·형식·예약 ID 검사를 통과해야 한다.
  - ZITADEL provider 로그인명과 호텔관리 canonical 로그인 ID는 복구 가능한 saga로 함께 변경하며, 한쪽만 변경된 상태를 성공으로 반환하지 않는다.
  - 검증된 ZITADEL subject와 호텔관리 내부 사용자 ID는 유지해 과거 업무·감사 주체를 보존한다.
  - 변경 성공과 동시에 해당 사용자의 모든 호텔관리 로그인 세션을 회수하고 다시 로그인하게 한다.
  - 이전 로그인 ID는 다른 사용자에게 영구 재사용하지 않는다.
  - 변경 요청·행위자·대상·이전 ID·새 ID·결과·시각을 감사하되 credential과 provider 오류 원문은 저장하지 않는다.
- 미확정:
  - 추가 재인증 또는 이중승인 필요 여부

### LOGIN-ID-007 로그인 ID 변경 전용 권한

- 상태: `confirmed`
- 질문: 어떤 관리자에게 로그인 ID 변경 권한을 부여할 것인가
- 사용자 답변: 별도 `USER_LOGIN_ID_CHANGE` 권한을 받은 관리자만 허용한다.
- 확정 규칙:
  - 로그인 ID 변경 API는 DB 정본의 회사범위 `USER_LOGIN_ID_CHANGE` capability를 요구한다.
  - `USER_CREATE`, `USER_SUSPEND`, 사용자유형, 직책, 표시이름, 정적 관리자 기본권한으로 변경권한을 암묵 부여하지 않는다.
  - 권한이 회수되면 정적 역할 기본값으로 복원하지 않고 즉시 변경 요청을 차단한다.
  - 행위자와 대상 사용자의 회사범위를 서버에서 검증한다.
- 영향:
  - capability registry, 권한 관리 UI, API 권한검사, 감사와 권한차단 테스트에 `USER_LOGIN_ID_CHANGE`를 추가해야 한다.
  - 재인증과 이중승인은 별도 정책 결정에 따른다.

### LOGIN-ID-008 로그인 ID 변경 재인증

- 상태: `confirmed`
- 질문: 로그인 ID 변경 시 추가 인증이나 이중승인을 요구할 것인가
- 사용자 답변: TOTP 재인증 후 즉시 변경하며 이중승인은 요구하지 않는다.
- 확정 규칙:
  - `USER_LOGIN_ID_CHANGE` 권한만으로 변경을 실행하지 않고 행위자의 TOTP 재인증을 요구한다.
  - TOTP 성공 뒤 행위자의 활성 session, 회사범위 권한, 대상 사용자, 새 ID 유일성·형식, 데이터 version을 서버에서 다시 검증한 뒤 변경한다.
  - 다른 관리자의 이중승인은 요구하지 않는다.
  - TOTP 미등록·실패·만료·복구중 상태에서는 변경하지 않는다.
  - TOTP code 원문은 DB·로그·감사·오류에 저장하지 않고 재인증 방법·결과·시각만 감사한다.
- 영향:
  - 일반 로그인은 기존 아이디·비밀번호 흐름을 유지하며 TOTP는 로그인 ID 변경 민감작업에서만 추가로 요구한다.
  - TOTP 분실·초기화는 기존 보안 복구정책을 따르며 로그인 ID 변경 API에서 우회수단을 만들지 않는다.

### LOGIN-ID-009 예약 로그인 ID 관리 방식

- 상태: `confirmed`
- 질문: 시스템용 로그인 ID를 일반 사용자에게 발급하지 못하게 어떻게 관리할 것인가
- 사용자 답변: 필수 예약 ID는 고정하고 회사별 추가 목록을 관리한다.
- 확정 규칙:
  - `admin`, `administrator`, `root`, `system`, `security`, `api`, `service`, `support`, `test`, `preview`, `werehere`는 필수 예약 ID로 고정한다.
  - 필수 예약 ID는 일반 사람 계정 생성·로그인 ID 변경에 사용할 수 없고 관리자가 삭제·해제할 수 없다.
  - 회사는 업무상 보호가 필요한 추가 로그인 ID를 별도 예약 목록에 등록할 수 있다.
  - 계정 생성과 로그인 ID 변경 시 canonicalize한 뒤 필수 목록과 회사 추가 목록을 모두 검사한다.
  - 이미 사용자에게 발급되었거나 과거 ID로 영구 예약된 값을 추가 예약 목록에 소급 등록하는 경우 충돌로 차단하고 자동으로 계정을 변경하지 않는다.
- 영향:
  - 필수 목록은 코드·계약 테스트의 최소 보안선으로 유지한다.
  - 회사별 추가 목록의 관리권한·재인증·삭제정책과 전역 ID 공간에서의 충돌 처리는 후속 정책으로 확정한다.

### LOGIN-ID-010 회사별 추가 예약 ID 관리권한

- 상태: `confirmed`
- 질문: 회사별 추가 예약 로그인 ID를 누가 관리할 것인가
- 사용자 답변: 별도 `USER_LOGIN_ID_RESERVATION_MANAGE` 권한과 TOTP 재인증을 요구한다.
- 확정 규칙:
  - 회사 추가 예약 ID의 등록·해제 API는 DB 정본의 회사범위 `USER_LOGIN_ID_RESERVATION_MANAGE` capability를 요구한다.
  - `USER_LOGIN_ID_CHANGE`, `USER_CREATE`, `USER_SUSPEND`, 사용자유형, 직책, 표시이름, 정적 역할 기본권한으로 관리권한을 암묵 부여하지 않는다.
  - 실제 등록·해제 전에 행위자의 TOTP 재인증을 요구하고, 성공 뒤 활성 session·회사범위 권한·목록 version·대상 ID 충돌을 다시 검증한다.
  - 필수 고정 예약 ID는 이 권한으로도 변경·해제할 수 없다.
  - TOTP 미등록·실패·만료·복구중 상태에서는 등록·해제하지 않는다.
  - 변경 전후 값·행위자·회사·결과·시각을 감사하되 TOTP code 원문을 저장하지 않는다.
- 영향:
  - capability registry, 권한 관리 UI, 예약 ID 관리 API·화면, 감사·권한차단·동시수정 테스트에 전용 capability를 추가해야 한다.
  - 추가 예약 ID의 해제 조건과 전역 ID 공간 충돌 처리는 별도 정책 결정에 따른다.

### LOGIN-ID-011 회사 추가 예약 ID 조건부 2단계 해제

- 상태: `confirmed`
- 질문: 회사가 추가로 예약한 로그인 ID를 나중에 어떻게 안전하게 해제할 것인가
- 사용자 답변: 싱드 권장 조건부 2단계 해제안을 적용한다.
- 확정 규칙:
  - 최종 등록 전에 canonicalization, 형식, 필수 예약, 현재·과거 사용자 ID, 기존 예약 충돌과 적용 영향을 미리 보여주고 TOTP 재인증 뒤 확정한다.
  - 최종 확정 전 초안은 효력이 없고 즉시 취소할 수 있다.
  - 활성 추가 예약 ID의 해제는 즉시 적용하지 않고 `RELEASE_PENDING`으로 전환해 7일간 계속 사용을 차단한다.
  - 7일 대기 중 `USER_LOGIN_ID_RESERVATION_MANAGE` 권한과 TOTP 재인증으로 해제 요청을 취소해 `ACTIVE`로 되돌릴 수 있다.
  - 대기기간 종료 시 필수 예약, 현재 사용자 ID, 비활성·과거 사용자 ID, 이전 로그인 ID tombstone, 다른 활성 예약, 취소 여부와 목록 version을 transaction 안에서 다시 검사한다.
  - 최종검사 통과 시 `RELEASED`, 충돌 시 `RELEASE_BLOCKED`로 전환한다. 해제된 이력도 물리삭제하지 않는다.
  - 현재 또는 과거 사용자에게 한 번이라도 발급됐거나 provider identity·과거 로그인 ID로 연결된 값은 `TOMBSTONED`로 영구 보호하고 해제할 수 없다.
  - 등록·해제요청·취소·최종해제·차단의 행위자·회사·대상 ID·전후상태·결과·시각을 감사한다.
  - 예약 검사와 계정 ID 발급은 canonical ID 기준 DB unique constraint와 같은 transaction 잠금·최종 재검사로 경쟁조건을 차단한다.
  - 초기 운영에서는 다른 관리자의 이중승인을 추가로 요구하지 않는다.
- 영향:
  - 예약 상태는 최소 `ACTIVE`, `RELEASE_PENDING`, `RELEASED`, `RELEASE_BLOCKED`, `TOMBSTONED`를 구분한다.
  - 7일 만료 작업은 멱등·재시도 가능해야 하고 최종검사 실패를 성공으로 기록하지 않는다.
  - 전역 ID 공간에서 회사 추가 예약이 다른 회사에 미치는 범위는 다음 정책으로 확정한다.

## 기록 원칙

- 사용자에게 다음 정책 질문을 제시하기 전에 직전 답변의 원장 반영 내용을 먼저 알린다.
- 사용자가 현재 질문의 자세한 설명을 요청하면 질문 진행과 원장 확정을 잠시 멈춘다.
- 설명 자체를 선택으로 간주하지 않으며, 사용자가 명시적으로 확정한 뒤에만 `confirmed`로 변경한다.
- 비밀번호, token, provider credential, 전체 provider 오류 본문은 이 원장에 기록하지 않는다.
