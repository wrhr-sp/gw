# 호텔관리 초기 MVP 상위 PRD

## 1. 문서 정보

| 항목 | 값 |
|---|---|
| PRD ID | `HOTEL-MVP-000` |
| 버전 | `1.0-user-approved` |
| 상태 | `user_approved` |
| 대상 | 위아히어 내부 호텔관리 |
| 승인자 | 대장, 2026-07-15 |
| 근거 | `COM-Q-001~060`, `HDRAFT-001~019` |

## 2. 목표

위아히어 법인 아래 내부 지점으로 관리되는 호텔에서 객실 기준정보, 객실점검, 일매출, 운영이슈, 호텔 소유주 문의를 실제 운영한다. 화면만 제공하지 않고 UI → same-origin API → Service/Repository → PostgreSQL/R2 → 권한·검증·감사 → 재조회까지 연결한다.

## 3. 해결할 문제

- 호텔별 객실점검 담당·결과·이상상태가 분산돼 추적이 어렵다.
- 매출 마감·환불·정정 증빙과 책임자가 일관되게 남지 않는다.
- 운영이슈의 담당·조치·종료와 호텔 소유주 공개범위가 분리되지 않는다.
- 호텔 소유주가 위탁사 문의를 남기고 답변 상태를 추적할 단일 창구가 없다.
- 지점배정만으로 민감기능 전체를 허용하거나 화면 숨김만으로 권한을 처리할 위험이 있다.

## 4. 사용자

| 사용자유형 | 초기 MVP 업무 |
|---|---|
| 사내 임직원 | 호텔배정 범위의 점검·이슈 기본업무, 추가권한에 따른 매출·설정·문의답변 |
| 호텔 소유주 | 자기 호텔 허용자료 조회, 확정매출 조회, 운영이슈 공개내용, 본인 문의 |
| 하우스키핑 | 배정 호텔 최소정보, 본인 담당점검, 이슈등록·본인 담당조치 |
| 거래처 임직원 | 호텔관리와 분리. 계정·로그인·별도 거래처 업무까지 확장 MVP로 이동 |

외부 사용자에게 사번·부서·급여·사내 개인정보를 부여하지 않는다. 하우스키핑은 개인별 독립 사용자유형이며 공용계정이 아니다.

## 5. 초기 MVP 범위

### 포함

1. 호텔 지점 생성·준비·활성화·운영중지·재활성화
2. 사내 임직원과 하우스키핑의 기간기반 호텔배정
3. 호텔 소유주 1:1 개인계정 연결과 안전한 사람 교체
4. 객실·객실유형·운영상태
5. 회사 공통 필수 체크리스트 + 호텔별 추가항목
6. 반복·수시 객실점검, 담당자, 사진·메모, 완료
7. 중대·긴급 이상항목의 운영이슈 자동연결
8. 운영이슈 상태·담당·조치·공개댓글·내부메모
9. 호텔 업무일별 일매출·확정·잠금·정정·증빙
10. 호텔 소유주 문의처와 인앱 문의티켓
11. 인앱 알림과 허용된 PWA 웹푸시
12. 비공개 R2, 설정형 파일 조회·다운로드 권한, 접근감사
13. 법인·호텔·사용자유형·배정·기능권한 기반 서버 권한검사

### 제외

- 근태·휴가·전자결재·일반 협업기능
- 거래처 임직원 계정·업무모듈
- PMS·채널매니저·OTA 실제연동
- 청소대기·청소중·청소완료 상태 및 청소배정 흐름
- SMS·알림톡·이메일 알림
- 대량 파일/자료 내보내기
- Production DB·secret·DNS·유료자원 변경

외부연동은 특정 사업자에 종속되지 않도록 외부 식별자·매핑·동기화 상태를 나중에 추가할 데이터 경계만 둔다. 가짜 연결 성공이나 fallback은 금지한다.

## 6. 공통 정책

- 데이터 하드경계: `company_id`
- 호텔범위: `branch_id` 또는 `hotel_id`
- 업무일: `Asia/Seoul` 기준 PostgreSQL `DATE`
- 사건시각: UTC `timestamptz`
- 종료일 UI 포함, 내부 범위 `[start, end)`
- 변경자료: `version` 낙관적 잠금, 오래된 저장은 `409`
- 모든 변경 API: 멱등키, 동일 키·동일 요청은 최초결과 재응답, 다른 요청은 충돌
- 멱등기록: 24시간, 요청·응답 원문 미저장
- 동일 DB 관련변경: 단일 transaction
- 외부작업: 상태저장·제한 재시도·보상·운영알림
- 오류: 안정된 코드·안전한 문구·필드오류·재시도정보·추적 ID 분리
- 사용·참조된 자료: 물리삭제 금지, 상태·정정·버전으로 보존

## 8. 화면 구조

- 목록: `PageHeader → FilterBar → DataTable → Pagination`

- 작성: `PageHeader → FormSection → ActionButtonGroup → ConfirmDialog`

- PC 우선 반응형, 모바일/PWA는 점검·사진·알림의 현장사용 흐름을 제공

## 9. 현재 구현사실과 구현격차

- 현재 `GET /api/branches`는 인증·가시범위 기반 읽기 전용이며 응답도 지점 생성·수정 저장이 범위 밖이라고 명시한다.
- `notifications`, `audit_logs`, `file_access_logs`, `permission_change_logs` 기반은 존재하지만 호텔 도메인 사건 연결은 아직 없다.
- `packages/shared/src/contracts.ts`에 호텔·객실·점검·매출·운영이슈·문의 계약과 `HOTEL_*` 권한코드가 아직 완결돼 있지 않다.
- 호텔관리 전용 Web route와 화면은 현재 확인되지 않았다.
- 신규 migration은 이미 사용 중인 `0043` 다음인 `0044` 이상이어야 한다.
- 따라서 본 문서는 기존 구현 완료를 주장하지 않으며 신규 full-stack 구현의 기준이다.

## 10. 상위 API 원칙

제안 경계:

```text
/api/hotels/*
/api/hotels/:hotelId/assignments/*
/api/hotels/:hotelId/rooms/*
/api/hotels/:hotelId/inspection-templates/*
/api/hotels/:hotelId/inspections/*
/api/hotels/:hotelId/issues/*
/api/hotels/:hotelId/daily-sales/*
/api/hotels/:hotelId/inquiries/*
/api/hotel-files/*
/api/admin/hotel-permissions/*
```

실제 경로는 `packages/shared/src/contracts.ts`의 `appRoutes`에 먼저 정의한다. 현재 `/api/branches`는 읽기 전용이므로 저장 API와 호텔 도메인 테이블은 신규 구현이 필요하다.
