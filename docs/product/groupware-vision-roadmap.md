# 한국형 그룹웨어 제품 비전 로드맵

## 1. 문서 목적

이 문서는 한국형 중소·중견 조직을 위한 그룹웨어 제품이 앞으로 어떤 문제를 풀고, 어떤 사용자에게 어떤 순서로 가치를 주고, 어떤 모듈을 우선 구현해야 하는지 한눈에 정리한 기준 문서입니다.

이번 문서는 다음 용도로 씁니다.

- 제품 방향을 기능 목록이 아니라 사용자 업무 흐름 기준으로 맞춘다.
- Preview/MVP 이후 어떤 순서로 고도화할지 3단계 로드맵으로 고정한다.
- UX, API, 권한, 문서 저장, 관리자 기능이 같은 원칙을 보도록 handoff 한다.
- 지금 하지 않을 일과 별도 승인 필요 범위를 미리 분리한다.

핵심 전제는 아래와 같습니다.

- Cloudflare-first, D1 우선, OpenNext on Cloudflare + Workers/Hono REST API 방향을 유지한다.
- 넓은 화면은 왼쪽 사이드바, 좁은 모바일 화면은 하단 탭 기본안을 유지한다.
- 국내 그룹웨어 공개 자료는 많이 참고하되, 특정 경쟁사의 문구/색상/레이아웃/화면을 복제하지 않는다.
- 이번 문서는 비전·로드맵·원칙·개발 handoff 까지만 다루며, 배포/DNS/custom domain/유료 리소스/production data migration 은 별도 승인 범위다.

## 2. 제품 비전

우리가 만들려는 제품은 "일반 직원은 오늘 해야 할 일을 빠르게 끝내고, 팀장/승인자는 병목 없이 처리하고, 인사/총무와 관리자는 정책과 운영 상태를 안전하게 관리하고, 대표/임원은 조직 운영 현황을 한눈에 보는" 한국형 업무 허브입니다.

조금 더 쉽게 말하면 아래 4가지를 한 제품 경험으로 묶는 것이 비전입니다.

1. 일상 업무 처리
   - 출근/퇴근
   - 휴가 신청
   - 전자결재 기안/승인
   - 공지 확인
   - 문서 조회

2. 팀 운영 정리
   - 승인 대기 정리
   - 팀 근태 예외 확인
   - 공지 전달 상태 확인
   - 자주 쓰는 문서/양식 접근

3. 인사/총무 운영 관리
   - 조직/직원/권한 관리
   - 근태/휴가 정책 관리
   - 결재 양식/게시판/문서 공간 관리
   - 감사 로그와 운영 예외 추적

4. 경영 가시성 확보
   - 조직 상태 요약
   - 승인 병목/지연 구간 파악
   - 근태/휴가/공지/문서 운영 현황 파악
   - 향후 급여/노무/자동화 확장 기반 확보

즉, 단순히 기능을 많이 넣는 제품이 아니라 "사람마다 오늘 필요한 업무를 가장 짧은 경로로 처리하게 하는 운영형 그룹웨어"를 목표로 합니다.

## 3. 핵심 사용자와 기대 가치

### 3-1. 일반 직원

주요 기대:

- 오늘 출근/퇴근, 휴가 신청, 결재 확인을 빠르게 끝내고 싶다.
- 내가 볼 수 있는 공지/문서만 깔끔하게 보고 싶다.
- 모바일에서도 핵심 업무를 무리 없이 처리하고 싶다.

제품이 줘야 하는 가치:

- 첫 화면에서 오늘 해야 할 일과 현재 상태를 바로 보여 준다.
- 상태 변경 CTA 를 찾기 쉽게 둔다.
- 권한 없는 메뉴나 헷갈리는 관리자 기능을 섞어 보여 주지 않는다.

### 3-2. 팀장 / 결재자

주요 기대:

- 승인 대기 건과 팀 예외를 빠르게 확인하고 처리하고 싶다.
- 팀원 요청의 문맥을 보고 실수 없이 승인/반려하고 싶다.
- 모바일에서도 급한 승인 정도는 바로 끝내고 싶다.

제품이 줘야 하는 가치:

- 승인함, 팀 대기, 예외 현황을 카드/목록으로 우선 노출한다.
- 승인 직전 필요한 상태 정보와 영향 정보를 같이 보여 준다.
- 승인/반려/보완 요청 액션의 실수 비용을 줄이는 UX 를 제공한다.

### 3-3. 인사 / 총무

주요 기대:

- 근태/휴가/결재/공지/문서 운영 기준을 관리하고 싶다.
- 조직 정책 변경과 예외 처리를 추적 가능하게 남기고 싶다.
- 운영 문의가 들어왔을 때 권한과 로그를 기준으로 설명하고 싶다.

제품이 줘야 하는 가치:

- 관리자 화면과 일반 업무 화면을 분리한다.
- 정책, 권한, 문서 공간, 게시판 운영을 한 흐름으로 관리하게 한다.
- 예외 처리와 감사 로그 후보를 남길 수 있게 한다.

### 3-4. 관리자

주요 기대:

- 사용자/조직/권한/정책/감사 로그를 안전하게 통제하고 싶다.
- 실제 운영 데이터와 설정 변경의 경계를 분명히 하고 싶다.
- 서비스 확장 시 API, 스토리지, 자동화 구조가 흔들리지 않길 바란다.

제품이 줘야 하는 가치:

- `/admin` 계열을 일반 사용자 흐름과 분리한다.
- same-origin API, D1, R2, 권한 검증 같은 기반 원칙을 유지한다.
- 운영 리소스 연결 전에도 문서와 skeleton 으로 확장 경로를 읽을 수 있게 한다.

### 3-5. 대표 / 임원

주요 기대:

- 조직 운영 상태를 요약해서 보고 싶다.
- 승인 지연, 근태 예외, 공지 전달, 문서 운영의 큰 흐름을 알고 싶다.
- 나중에 급여/노무/자동화까지 이어질 수 있는지 판단하고 싶다.

제품이 줘야 하는 가치:

- 대시보드에서 KPI 성격의 운영 요약을 제공한다.
- 세부 화면을 다 열지 않아도 병목과 리스크를 이해하게 한다.
- 향후 운영 자동화와 경영 리포트 확장 가능성을 남긴다.

## 4. 제품 원칙

### 원칙 A. 기능보다 업무 흐름을 먼저 설계한다.

상위 메뉴는 "무슨 기능이 있나"보다 "사용자가 지금 무슨 일을 끝내려 하나" 기준으로 묶습니다.

권장 1차 업무 묶음은 아래와 같습니다.

- 홈/대시보드
- 근태/휴가
- 전자결재
- 게시판/공지
- 문서/파일
- 조직/직원
- 관리자

여기서 `조직/직원`은 관리자 메뉴 안에 접어 넣는 하위 기능이 아니라 `/org`, `/employees` 로 직접 진입할 수 있는 독립 1차 탐색/업무 모듈입니다. 다만 모바일 하단 탭의 기본 4~6개 항목에는 올리지 않고, 데스크톱 사이드바의 조직/관리 그룹 또는 홈/대시보드 진입점에서 역할별로 노출합니다. `/admin/*` 은 정책, 권한, 감사 로그, 운영 설정을 통제하는 관리자 capability 로 분리합니다.

### 원칙 B. 첫 화면은 오늘 처리할 일을 먼저 보여 준다.

홈/대시보드는 정보 창고가 아니라 행동 시작점이어야 합니다.

우선순위는 아래 순서를 기본으로 둡니다.

1. 지금 바로 해야 하는 액션
2. 승인/대기/예외 상태
3. 최근 기록 요약
4. 자주 가는 업무 진입점
5. 정책/안내/참고 링크

### 원칙 C. UX 는 데스크톱과 모바일의 껍데기만 다르게 한다.

- 데스크톱/태블릿: 왼쪽 사이드바
- 모바일 좁은 화면: 하단 탭
- 같은 정보구조와 route 를 유지하고, 탐색 UI 만 화면 크기에 맞게 바꾼다.

### 원칙 D. 권한 경계를 화면부터 명확하게 드러낸다.

- 관리자 기능은 일반 사용자 하단 탭에 섞지 않는다.
- 권한 없는 액션은 숨기거나 비활성화하되 이유를 설명한다.
- 상태 변경 API 는 화면 제어와 별개로 서버 권한 검증을 유지한다.

### 원칙 E. 빈 상태와 오류는 친절한 포장보다 제약 설명이 먼저다.

- 왜 비어 있는지
- 지금 사용자가 할 수 있는 일
- 무엇은 아직 안 되는지
- 다시 시도할 방법이 무엇인지

이 순서로 안내합니다.

### 원칙 F. 제품 기본 URL 원칙은 same-origin 을 유지한다.

- Web/PWA 기본 경로는 현재 열려 있는 origin 기준이다.
- API 기본 경로는 `/api/*` same-origin 상대 경로를 우선한다.
- manifest 는 `/manifest.webmanifest`, `start_url` 은 `/` 를 유지한다.
- preview 전용 절대 API 도메인을 제품 기본값으로 박아 넣지 않는다.

## 5. 3단계 제품 로드맵

## 단계 1. Preview / MVP

목표:

- 제품의 핵심 업무 흐름과 정보구조를 작동 가능한 skeleton 수준으로 맞춘다.
- 다음 구현자가 바로 이어서 작업할 수 있는 API, shared contract, D1 migration, Web placeholder, 문서 기준을 고정한다.

포함 초점:

- 로그인/권한 기본 골격
- 조직/직원 기본 조회와 회사/직원 scope 골격
- 대시보드 기본 구조
- 근태/휴가 1차 흐름
- 전자결재 1차 흐름
- 게시판/문서 1차 흐름
- 모바일/PWA 기본 구조
- same-origin API 연결 기준

이 단계의 성공 기준:

- 사용자가 업무 묶음 구조를 이해할 수 있다.
- `/org`, `/employees` 가 독립 업무 모듈로 보이고, `/admin/*` 운영 설정과 섞이지 않는다.
- 각 핵심 모듈이 placeholder 또는 skeleton 이라도 같은 규칙으로 연결된다.
- 문서와 코드가 같은 제품 방향을 가리킨다.
- 운영 승인 없이 production 리소스를 건드리지 않는다.

## 단계 2. 업무 연결 고도화

목표:

- 기능을 따로따로 두지 않고, 실제 업무 연결과 역할 기반 처리 흐름을 강화한다.

포함 초점:

- 승인과 근태/휴가의 연결
- 게시판/문서와 공지 전달 흐름 연결
- 조직/직원 업무 모듈과 권한/정책/관리자 설정 연결
- 감사 로그와 운영 예외 추적 강화
- 문서 공간, 첨부 metadata, 접근 제어 보강
- 팀장/결재자용 병목 해소 UX 보강

이 단계의 성공 기준:

- 팀장/승인자가 대기 업무를 더 짧은 클릭 수로 처리한다.
- 인사/총무가 정책/권한/예외를 한 흐름에서 관리한다.
- 인사/총무와 팀장이 `/org`, `/employees` 에서 조직 구조와 직원 기본 정보를 먼저 확인한 뒤, 필요한 운영 설정만 `/admin/*` 로 이동한다.
- 게시판/문서/결재/근태 데이터가 고립되지 않고 업무 문맥으로 이어진다.

## 단계 3. 운영 / 자동화 / 모바일 확장

목표:

- 운영자가 반복 업무를 줄이고, 경영층은 더 높은 수준의 운영 가시성을 얻고, 모바일에서도 핵심 흐름을 안정적으로 끝내게 한다.

포함 초점:

- 알림/자동화/예약 작업
- 모바일/PWA 고도화
- 관리자 운영 리포트/감사 흐름 고도화
- 향후 급여/노무, 자동 계산, 확장 API 연계 준비
- R2, Queue, Cron, Durable Objects 같은 운영 확장 포인트 정리

이 단계의 성공 기준:

- 반복 확인/리마인드/예외 추적 업무가 줄어든다.
- 모바일에서 자주 쓰는 업무의 완료율이 높아진다.
- 운영 데이터와 정책 변경 이력이 더 명확히 추적된다.

## 6. 모듈 우선순위

### 1순위. 로그인 / 권한

이유:

- 모든 업무 흐름의 시작점이다.
- 권한 경계를 먼저 세우지 않으면 관리자/일반 사용자 경험이 섞인다.
- 이후 모든 API 와 화면 규칙의 공통 기반이다.

핵심 범위:

- 로그인/로그아웃
- 세션/내 정보
- 역할/권한 확인
- 회사/조직 scope
- 관리자 접근 경계

### 2순위. 조직 / 직원

판정:

- `조직/직원`은 `/org`, `/employees` 로 직접 진입하는 독립 1차 탐색/업무 모듈이다.
- 관리자/권한 모듈 내부 capability 로 접지 않는다.
- 단, 직원 생성/권한 변경/정책 변경/감사 로그 같은 운영 통제 액션은 `/admin/*` 에 남긴다.

이유:

- 회사/조직 scope, 직원 기준 데이터, 역할 표시가 이후 근태/휴가/결재/게시판/문서 권한의 공통 기반이다.
- 인사/총무와 팀장은 관리자 설정에 들어가기 전에도 조직도와 직원 기본 정보를 업무 화면에서 확인해야 한다.
- `docs/architecture/next-cloudflare-platform-plan.md` 의 초기 route(`/employees`, `/org`)와 `docs/ux/groupware-benchmark-principles.md` 의 조직/관리 IA 원칙을 그대로 따른다.

핵심 범위:

- `/org`: 부서/팀/직책/상하 관계를 읽는 조직도 skeleton
- `/employees`: 직원 목록, 직원 기본 프로필, 소속/직책/상태 표시 skeleton
- 회사/직원 scope 와 역할 표시
- 권한별 조회/관리 액션 노출 경계
- `/admin/users`, `/admin/policies`, `/admin/audit-logs` 로 넘어가야 하는 운영 통제 액션 분리

성공 기준:

- 일반 사용자/팀장/인사/관리자 모두 같은 `/org`, `/employees` route 를 보되 권한에 따라 보이는 필드와 액션이 다르다.
- 직원 조회와 조직 구조 확인은 독립 업무 화면에서 가능하고, 권한 변경·정책 변경·감사 로그는 `/admin/*` 로만 이어진다.
- 모바일에서는 하단 탭 기본 항목이 아니라 대시보드 바로가기, 더보기, 화면 내부 링크로 진입한다.

### 3순위. 대시보드

이유:

- 제품 비전을 실제 사용 경험으로 보여 주는 첫 화면이다.
- "오늘 처리할 일" 원칙이 가장 먼저 드러나야 한다.

핵심 범위:

- 오늘 해야 할 액션
- 승인/대기/예외 요약
- 최근 기록
- 자주 가는 업무 바로가기

### 4순위. 근태

이유:

- 사용 빈도가 높고 모바일 가치가 크다.
- 이후 휴가/급여/노무 확장의 기반이 된다.

핵심 범위:

- 출근/퇴근
- 근태 기록 조회
- 정정 요청
- 관리자/승인자 예외 확인

### 5순위. 휴가

이유:

- 근태와 가장 밀접하게 연결된다.
- 직원/팀장/인사 모두 사용하는 공통 업무다.

핵심 범위:

- 휴가 유형
- 잔여 현황
- 신청/승인/반려
- 정책 안내와 예외 처리

### 6순위. 전자결재

이유:

- 팀장/결재자 관점의 병목 해소 가치가 크다.
- 게시판/문서/근태/휴가와 연결되는 공통 승인 허브 역할을 한다.

핵심 범위:

- 양식
- 기안
- 승인함
- 반려/보완
- 결재선/참조

### 7순위. 게시판

이유:

- 공지 전달과 읽음 확인 같은 조직 커뮤니케이션의 기본이다.
- 사용자가 매일 접속하는 이유를 만드는 모듈이다.

핵심 범위:

- 공지/게시판 목록
- 게시글 상세
- 댓글/읽음 확인
- 권한별 작성/관리 경계

### 8순위. 문서

이유:

- 게시판과 함께 협업 맥락을 만든다.
- 접근 권한, 첨부 metadata, 보관 규칙이 중요하다.

핵심 범위:

- 문서 공간
- 문서 목록
- 첨부 metadata
- 다운로드/접근 제어

### 9순위. 관리자

이유:

- 앞선 모듈이 실제 운영 가능한 상태로 이어지게 하는 제어판이다.
- 일반 사용자 UX 를 해치지 않으면서 운영 통제를 제공해야 한다.
- `조직/직원` 업무 화면에서 하지 않는 사용자 생성, 권한 변경, 정책 변경, 감사 확인을 맡는다.

핵심 범위:

- 사용자/조직/권한 관리
- 정책 관리
- 게시판/문서 공간 관리
- 감사 로그
- 운영 예외 확인

### 10순위. API / D1 / R2 기반 확장

이유:

- 화면보다 뒤에 숨지만, 장기적으로 운영 일관성과 자동화 확장의 기반이다.
- same-origin API, D1 binding, 문서 저장, 자동화, 알림 확장을 안정적으로 받쳐야 한다.

핵심 범위:

- same-origin `/api/*`
- D1 schema/migration/binding
- 문서/R2 저장소 연결
- Queue/Cron 기반 자동화 후보
- 운영/감사 데이터 구조

## 7. UX 원칙 요약

개발자와 디자이너가 바로 기억해야 할 핵심은 아래 5가지입니다.

1. 넓은 화면은 왼쪽 사이드바, 좁은 화면은 하단 탭을 유지한다.
2. 업무 CTA 는 카드 맨 아래가 아니라 상태 요약 바로 옆이나 아래에 둔다.
3. 빈 상태/오류/오프라인 안내는 "왜 이런 상태인지 + 지금 뭘 해야 하는지"를 먼저 보여 준다.
4. 권한에 따라 메뉴/버튼/상세 액션 노출을 다르게 한다.
5. 모바일은 축소판이 아니라 핵심 행동 우선 재배치 버전으로 설계한다.

더 자세한 UX 기준은 `docs/ux/groupware-benchmark-principles.md` 를 먼저 본다.

## 8. Phase 7+ 연결 원칙

이 문서 기준으로 Phase 7+ 확장은 아래 순서를 지켜야 합니다.

1. API same-origin 원칙 유지
   - Web/PWA 는 계속 현재 origin 아래 `/api/*` 를 기본값으로 본다.

2. D1 binding 과 데이터 구조 정합성 유지
   - 업무 모듈을 늘릴수록 D1 migration 과 company/employee/role scope 를 더 엄격히 본다.

3. 인증/권한 규칙 재사용
   - 새 모듈이 생겨도 권한은 화면에서만 가리지 말고 API 에서 다시 검증한다.

4. 관리자 접근 제어 분리
   - 급여/노무, 정책, 감사 기능은 일반 사용자 흐름에 섞지 않는다.
   - `조직/직원` route 는 팀장/인사/관리자 공용 조회 흐름으로 유지하되, 직원 생성·권한 변경·정책 변경·감사 로그 같은 운영 액션은 `/admin/*` 계열에 둔다.

5. 문서/R2 확장 시 접근 제어 우선
   - 파일 저장보다 먼저 문서 공간, 첨부 metadata, 다운로드 권한 설명이 맞아야 한다.

6. 알림/자동화는 운영 통제와 함께 도입
   - Queue/Cron/알림은 단순 자동 발송보다 예외 추적, 재시도, 감사 가능성을 같이 설계한다.

즉, Phase 7+ 는 "기능 추가"가 아니라 "기존 업무 허브를 운영형 플랫폼으로 키우는 단계"로 이해해야 합니다.

## 9. 이번 단계에서 하지 않을 것

이번 문서와 후속 구현에서 바로 하지 않을 것은 아래와 같습니다.

- 특정 경쟁사 화면/문구/색상/레이아웃 복제
- 로그인 뒤 비공개 화면이나 유료 데모 내부 화면 참고
- production 데이터 입력 또는 production DB migration 실행
- custom domain, DNS, 외부 공개 확대를 자동으로 진행하는 일
- 유료 리소스 생성이나 비용 발생 작업
- 실제 급여 계산, 세무/노무 실처리 로직 확정
- secret 출력, 커밋, 로그 노출

   - 이 문서의 2장, 3장, 5장을 먼저 읽고 누구의 어떤 문제를 어떤 순서로 푸는지 맞춘다.

2. UX 기준 확인

   - 데스크톱 사이드바 / 모바일 하단 탭 / CTA 위치 / 권한 노출 원칙 확인

3. 플랫폼 기준 확인
   - `docs/architecture/next-cloudflare-platform-plan.md`

   - `docs/architecture/phase-7-api-same-origin-scope.md`

4. 구현 우선순위 유지
   - 로그인/권한 → 조직/직원 → 대시보드 → 근태 → 휴가 → 전자결재 → 게시판 → 문서 → 관리자 → API/D1/R2 확장
   - `조직/직원`은 `/org`, `/employees` 독립 route 로 구현하고, `/admin/*` 은 운영 통제 액션만 맡긴다.

5. 조직/직원 구현 기준 먼저 맞추기
   - `/employees` 는 employee 기본 shape, 재직 상태, 소속 정보를 읽는 기준 route 로 본다.
   - `/org` 는 부서/팀/직책/상하 관계를 읽는 조직도 기준 route 로 본다.
   - 두 route 는 조회와 업무 문맥 확인 중심의 독립 화면으로 유지하고, 직원 생성/권한 변경/정책 변경/감사 로그 같은 운영 통제 액션은 `/admin/*` 과 분리한다.
   - 모바일 1차에서는 하단 탭 고정 진입보다 대시보드 바로가기, 더보기, 화면 내부 링크로 연결한다.

6. 품질 기준 유지
   - preview/prod 기본값에 절대 API 도메인 박지 않기
   - 상대 경로 manifest 와 same-origin `/api/*` 유지
   - 권한 없는 상태를 화면과 API 모두에서 분리
   - placeholder 는 실제 운영 완료처럼 보이지 않게 쓰기

## 11. Definition of Done

이 문서는 아래를 만족하면 제품 비전 로드맵 기준 문서로 쓸 수 있습니다.

- 핵심 사용자 5종의 기대 가치가 분리되어 있다.
- 3단계 로드맵이 Preview/MVP → 업무 연결 고도화 → 운영/자동화/모바일 확장 순서로 정리되어 있다.
- 모듈 우선순위와 이유가 같이 적혀 있다.
- `조직/직원`이 `/employees`, `/org` 기준의 독립 조회 업무 묶음으로 한 가지로만 해석되고, `/admin/*` 운영 통제 액션과 분리되어 있다.
- `/org`, `/employees`, `/admin/*` 의 역할 경계와 구현 순서가 handoff 에 명시되어 있다.
- UX 원칙과 same-origin/API/D1/R2 기반 원칙이 충돌 없이 연결되어 있다.
- 하지 않을 일과 승인 필요 범위가 분리되어 있다.
- 다음 구현자가 읽어야 할 문서와 순서가 handoff 에 정리되어 있다.
