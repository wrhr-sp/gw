# 관리자 계정 생성·계정 관리 개발 기준

## 목적
관리자페이지의 계정 생성 및 계정 관리는 단순 로그인 계정 생성이 아니라 회사 그룹웨어의 사용자 생명주기, 조직 연결, 역할/권한, 보안정책, 감사로그를 함께 다루는 운영 기능이다.

## 1차 적용 범위
- 관리자페이지 사이드바는 관리자 업무만 노출한다.
- 관리자페이지 사이드바에는 `/admin/users` 진입 버튼을 `사원 계정 관리`로 노출하고, 해당 기능 안에서 계정 생성/관리 흐름을 운영한다.
- 계정 목록, 계정 생성 검증, 역할/권한 변경 검토, 상태 변경 검토, 비밀번호 초기화 검토를 한 흐름으로 제공한다.
- 계정생성은 이름, 로그인 ID/이메일, 부서, 지점, 직책/직급, 계정 유형, 초기 역할, 초기 상태, 생성 사유, 보안 설정을 입력받고 `users → employees → user_roles → audit_logs → 재조회` 순서로 운영 DB에 저장한다.
- DB, schema, 역할, 부서, 지점 연결이 준비되지 않은 작업은 성공처럼 보이지 않고 명확한 오류 상태로 둔다.
- 일반 직원 화면, 부서업무포털, 지점관리포털, 경영업무포털성 기능과 관리자 업무를 섞지 않는다.

## 필수 계정 속성
- 계정 유형: 직원, 관리자, 외부 사용자, 봇/서비스 계정, 시스템 계정
- 계정 상태: 초대대기, 활성, 잠금, 비활성, 퇴사처리, 일시정지
- 조직 연결: 회사, 부서, 지점, 직책/직급, 직원 레코드 연결 상태
- 보안 상태: 최초 비밀번호 변경 필요, 2단계 인증 필요, 최근 로그인, 실패 횟수, 세션 수
- 권한 상태: 역할, 직접 권한 예외, 고위험 권한 여부, 관리자 scope

## 보안 원칙
- 비밀번호, 초대 토큰, 재설정 토큰, refresh token, API token 원문을 저장하거나 출력하지 않는다.
- 토큰류는 hash/digest만 저장한다.
- 계정 생성, 권한 변경, 상태 변경, 잠금/해제, 비밀번호 초기화, 세션 종료는 감사로그 후보 또는 실제 감사로그로 남긴다.
- production DB 실데이터, secret, 외부 IdP/SSO/LDAP, 실제 메일 발송, 유료 리소스는 별도 승인 게이트다.

## UI/UX 원칙
- 관리자 사이드바는 `관리자 업무`로 묶고 관리자 기능만 보여 준다.
- 일반 업무 메뉴는 관리자 사이드바에 섞지 않는다.
- 검색/필터 영역은 Grid 기반으로 균등 정렬한다.
- 버튼 묶음은 부모 toolbar 안에서 `display:flex`, `align-items:center`, `gap:8px`를 따른다.
- 계정 생성·저장·상태 변경 버튼은 실제 API가 없으면 내부 검증 또는 비활성/승인필요 상태로 둔다.

## API/백엔드 기준
- 목록 조회는 실제 DB 기반으로 하고 DB 미설정 시 `DB_NOT_CONFIGURED`로 안전 실패한다.
- mutation은 실제 Service/Repository/DB/audit 연결 전까지 2xx fake success를 반환하지 않는다.
- 계정 상태 enum과 계정 유형 enum은 shared contract에 둔다.
- API는 company scope, role/permission guard, forbidden/not found/validation/schema drift를 명확히 구분한다.

## 완료 기준
- 관리자 권한 계정은 `/admin/users`에서 계정관리 화면을 볼 수 있다.
- 일반 계정은 관리자 메뉴와 관리자 route/API 접근이 차단된다.
- 관리자 사이드바에는 관리자 업무만 나온다.
- no-DB 상태에서 fake success가 없다.
- 테스트, typecheck, Cloudflare build, PR/CI/merge/deploy/live smoke 근거가 있어야 완료로 본다.
