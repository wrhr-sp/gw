# 그룹웨어 외부 API·서버 보안 우선순위 결정안

## 1. 문서 목적

이 문서는 위아히어 자체 그룹웨어에 붙일 외부 API와 서버 보안 구성을 먼저 정리하기 위한 기준 문서다.

이번 단계의 핵심은 부서별 업무 메뉴를 바로 확정하는 것이 아니라, 그룹웨어가 계약·승인·문서·증빙·알림·보안 흐름을 안전하게 담을 수 있도록 외부 연동 우선순위와 보안 기반을 먼저 고정하는 것이다.

## 2. 현재 결정한 방향

### 2-1. API 우선순위의 큰 기준

자체 그룹웨어에 붙일 API는 단순 편의 기능보다 아래 업무를 보강하는 API를 우선한다.

1. 법적 증빙
2. 승인과 계약
3. 문서 보관
4. 세무·정산 증빙
5. 외부 문의/연락 수집
6. 신원 확인
7. 위치·현장 운영
8. OCR·AI 업무보조

즉, 그룹웨어를 단순 게시판이 아니라 계약·결재·문서·증빙·보안이 연결된 업무 시스템으로 만드는 API부터 붙인다.

### 2-2. 카카오 알림톡 / SMS의 결정 변경

카카오 알림톡 / SMS API는 단순 보류가 아니다.

다만 초기 용도는 `내부 결재 알림`만으로 한정하지 않는다. 위아히어 기준에서는 고객사·거래처·외부 파트너가 카카오톡이나 문자로 문의/연락한 내용을 그룹웨어로 받아오는 외부 커뮤니케이션 연동 후보로 본다.

예상 방향:

- 고객사 또는 거래처 문의 수신
- 지점/사업장 문의 수신
- 계약·정산·증빙 관련 외부 문의 접수
- 문의 내용으로 그룹웨어 업무 생성
- 담당자 배정
- 처리 상태 추적
- 필요 시 회신 이력 보관

따라서 카카오/SMS 연동은 `내부 알림 발송 API`가 아니라 `외부 연락 접수 + 업무화 + 추적` 기능으로 별도 설계한다.

## 3. API 우선순위 결정안

### 1순위. 전자계약 API

후보:

- 모두싸인
- 글로싸인
- 도뉴
- Adobe Sign

우선 이유:

- 근로계약서, 용역계약서, 위탁운영계약서, NDA, 입점계약서, 거래처 계약서에 바로 연결된다.
- 그룹웨어의 전자결재/승인 흐름과 가장 자연스럽게 붙는다.
- 서명 완료 이벤트를 Webhook으로 받아 내부 상태를 자동 변경할 수 있다.

필요 기능:

- 계약서 자동 생성
- 내부 승인 후 외부 서명 발송
- 서명 완료 Webhook 수신
- 계약서 PDF 자동 보관
- 계약 만료일 알림
- 계약 담당자·승인자 감사로그 저장

위아히어 적용 예:

- 입사 승인 → 근로계약서 자동 생성 → 전자계약 발송 → 서명 완료 Webhook 수신 → 인사기록 갱신 후보 생성

### 2순위. 파일 저장 API

후보:

- Cloudflare R2
- AWS S3
- NCP Object Storage

우선 이유:

- 계약서, 증빙, 인사서류, 시설사진, 보고서 파일을 DB에 직접 넣으면 안 된다.
- 파일 본문은 Object Storage에 두고, 그룹웨어 DB에는 파일 ID, 권한, 해시, 작성자, 보관기간, 감사로그만 둬야 한다.
- 현재 그룹웨어는 Cloudflare-first 방향이므로 R2가 가장 자연스럽다.

필요 기능:

- 파일 업로드/다운로드
- 권한별 접근 제어
- 임시 다운로드 URL
- 파일 보존기간 관리
- 문서 버전관리
- 개인정보 파일 접근로그
- 파일 해시 저장

위아히어 적용 예:

- 전자계약 PDF는 R2에 저장 → 그룹웨어에는 문서번호, 작성자, 승인자, 접근권한, 열람로그만 저장

### 3순위. 전자세금계산서 / 홈택스 연동 API

후보:

- CODEF
- Popbill
- Barobill
- 홈택스 연동 대행 API

우선 이유:

- 비용결의, 거래처 정산, 세무사 전달자료 자동화에 바로 필요하다.
- 직접 홈택스를 다루기보다 인증/조회/매칭을 제공하는 전문 API를 쓰는 편이 현실적이다.

필요 기능:

- 매입/매출 세금계산서 조회
- 거래처별 계산서 매칭
- 증빙 누락 감지
- 비용결의서와 세금계산서 연결
- 세무사 전달자료 자동 정리

위아히어 적용 예:

- 비용결의 등록 → 사업자번호 기준 세금계산서 조회 → 증빙 자동 매칭 → 월말 전달자료 생성

### 4순위. 본인인증 / 전자서명 인증 API

후보:

- NICE
- PASS 인증서
- 계좌점유인증 API

우선 이유:

- 외부 사용자, 협력업체, 사업장 관리자, 지점 담당자에게 권한을 주려면 신원 확인이 필요하다.
- 중요 결재나 개인정보 접근에는 2차 인증이 필요할 수 있다.

필요 기능:

- 외부 사용자 본인확인
- 계약 서명자 신원 확인
- 직원 계정 최초 등록 인증
- 급여계좌 등록 시 계좌점유인증
- 중요 결재 시 재인증

위아히어 적용 예:

- 사업장 관리자 계정 생성 → 휴대폰 본인확인 → 권한 부여 → 개인정보 접근 시 재인증

### 5순위. 외부 문의/연락 연동 API

후보:

- 카카오 비즈메시지/상담/채널 연동 후보
- SMS 수신/발신 API
- 이메일 수신 API
- 향후 채널톡, 카카오 채널, CRM API 후보

우선 이유:

- 카카오 알림톡/SMS를 단순 내부 알림 발송으로 쓰기보다, 고객사·거래처 연락을 그룹웨어 업무로 끌어오는 기능이 위아히어에 더 실용적이다.
- 문의가 카톡/문자/메일로 흩어지면 담당자 배정, 처리상태, 증빙 보관이 어려워진다.

필요 기능:

- 외부 문의 수신
- 발신자/거래처 매칭
- 문의 유형 분류
- 담당자 배정
- 업무 카드 생성
- 처리 상태 추적
- 답변/회신 이력 보관
- 계약·정산·시설·운영 이슈와 연결

위아히어 적용 예:

- 거래처가 카카오톡으로 정산 문의 → 그룹웨어 외부문의함 생성 → Revenue팀 또는 영업관리팀 담당자 배정 → 처리 완료 후 이력 보관

### 6순위. 이메일 / 캘린더 / 드라이브 API

후보:

- Google Workspace API
- Microsoft Graph API

우선 이유:

- 메일, 캘린더, 파일 저장소는 외부 업무도구와 함께 쓰는 경우가 많다.
- 자체 그룹웨어가 모든 메일/캘린더 기능을 독자 구현하기보다 외부 업무도구와 연동하는 편이 효율적이다.

필요 기능:

- 공용메일 수신 → 업무 생성
- 회의 일정 자동 등록
- 계약 만료 캘린더 등록
- 첨부파일 자동 저장
- 외부 업무 접수

위아히어 적용 예:

- 지원사업 메일 수신 → 첨부파일 저장 → 담당자 태스크 생성 → 마감일 캘린더 등록

### 7순위. OCR / 문서 인식 API

후보:

- CLOVA OCR
- Google Document AI
- AWS Textract
- Upstage OCR

우선 이유:

- 영수증, 세금계산서, 계약서, 사업자등록증, 통장사본 등 수기 입력이 많은 문서를 줄일 수 있다.

필요 기능:

- 영수증 금액 자동 인식
- 사업자등록증 정보 추출
- 통장사본 계좌정보 추출
- 계약서 주요항목 추출
- PDF 문서 자동 분류
- 첨부파일 누락 여부 확인

### 8순위. 지도 / 주소 / 사업장 위치 API

후보:

- 카카오맵 API
- 네이버 지도 API
- 도로명주소 API

우선 이유:

- 호텔 사업장, 지사, 권역, 시설, 현장 대응이 있는 위아히어 업무에 위치 정보가 중요하다.

필요 기능:

- 사업장 주소 표준화
- 지사/권역별 지도
- 현장 출동 거리 계산
- 시설관리 위치 태깅
- 주소 자동완성

### 9순위. 결제 / 정산 API

후보:

- 토스페이먼츠
- KG이니시스
- 나이스페이
- KCP

우선 이유:

- 그룹웨어 초기에는 결제보다 정산자료 관리가 우선이다.
- 다만 서비스비, 예약금, 관리비, OTA/MRO 정산 확장 시 필요해질 수 있다.

필요 기능:

- 결제 승인/취소 조회
- 매출전표/현금영수증 확인
- 거래별 정산 상태 추적
- 미수금 알림

### 10순위. AI API / 내부 에이전트 API

후보:

- OpenAI API
- 자체 Hermes Orchestrator API
- 문서요약 API

우선 이유:

- 자동 의사결정보다 초안·요약·분류·체크리스트 생성부터 붙이는 것이 안전하다.

필요 기능:

- 결재 요청 요약
- 계약서 주요 조건 추출
- 비용 증빙 이상 탐지
- 인사 문서 초안 생성
- 사고보고서 정리
- 매뉴얼 기반 질의응답

## 4. 1차 API 세트

초기 실제 연동 후보는 아래 5개로 본다.

1. 전자계약 API
2. R2/S3 파일 저장 API
3. 전자세금계산서/홈택스 연동 API
4. 본인인증/PASS/NICE API
5. 외부 문의/연락 연동 API

카카오 알림톡/SMS는 이 5번의 하위 후보로 두되, `내부 알림 발송`보다 `외부 고객사·거래처 문의 수신/업무화`를 먼저 설계한다.

## 5. 보안 구성 원칙

보안은 보안프로그램을 많이 붙이는 방식이 아니라 계층별로 나눈다.

기준 계층:

1. 사용자 PC / 모바일
2. 접속 구간
3. 그룹웨어 서버
4. 애플리케이션
5. DB
6. 파일 저장소
7. 모니터링 / 로그
8. 백업 / 복구

그룹웨어는 인사정보, 근태, 계약서, 급여자료, 증빙, 사업장 정보를 처리할 수 있으므로 개인정보처리시스템 기준으로 설계한다.

## 6. 보안 우선순위 결정안

### 1순위. 권한 테이블과 접근통제

가장 먼저 개발 구조에 반영해야 한다.

필요 항목:

- RBAC 기반 권한관리
- 부서/직책/역할별 접근권한
- 겸직 부서 접근권한
- 사업장/지점별 접근권한
- 관리자 계정과 일반 계정 분리
- 개인정보 접근권한 분리
- 권한 부여/회수 감사로그

### 2순위. 감사로그와 개인정보 접속기록

필수 로그:

- 로그인 성공/실패
- 로그아웃
- 비밀번호 변경
- MFA 실패
- 계정 잠금
- 개인정보 조회/수정/삭제
- 급여자료 조회
- 계약서 열람
- 파일 다운로드
- 대량 조회/다운로드
- 엑셀 내보내기
- 관리자 설정 변경
- 권한 변경
- 결재 승인/반려

기본 보관 기준:

- 일반 감사로그: 최소 1년
- 개인정보 접속기록: 최소 1년, 가능하면 2년
- 계약/급여/인사 중요 로그: 3~5년 권장

### 3순위. 파일 업로드 보안

필수 정책:

- 실행파일 업로드 금지
- 확장자 검증
- MIME 타입 검증
- 파일명 랜덤화
- 원본 파일명 별도 저장
- Object Storage 저장
- 다운로드 권한 확인
- 다운로드 로그 저장
- 악성코드 검사 후보
- 파일 해시 저장

금지 확장자 예:

- `.exe`
- `.bat`
- `.cmd`
- `.sh`
- `.ps1`
- `.jsp`
- `.php`
- `.asp`
- `.aspx`
- `.js`
- `.html`
- `.svg`

### 4순위. DB 보안

필수 정책:

- 운영 DB / 개발 DB 분리
- 개발 DB에 실제 개인정보 금지
- DB 외부 포트 차단
- DB 계정 최소권한
- 관리자 계정 분리
- DB 접속 로그 저장
- 중요 컬럼 암호화
- 백업 암호화
- 백업 복구 테스트

암호화 후보:

- 주민등록번호
- 외국인등록번호
- 계좌번호
- 급여자료
- 연락처
- 주소
- 신분증 사본
- 계약서 파일
- 4대보험 관련 자료

### 5순위. 백업 / 복구 / 랜섬웨어 대비

필수 정책:

- 일일 DB 백업
- 주간 백업
- 월 1회 복구 테스트
- 백업 암호화
- 백업 계정 분리
- 백업 삭제 권한 제한
- Object Lock 또는 버전관리 후보
- 운영 서버가 백업 저장소를 마음대로 삭제하지 못하게 분리

백업 대상:

- 운영 DB
- 파일 저장소 메타데이터
- 첨부파일
- 환경변수/설정값
- 배포 버전
- 감사로그
- 인증/권한 테이블

### 6순위. WAF / Rate Limit / 접속 보호

초기 추천:

- Cloudflare WAF
- Rate Limit
- `/admin`, `/api/internal`, `/debug` 등 민감 경로 보호
- 비정상 로그인 시도 차단
- 파일 업로드 공격 차단
- SQL Injection / XSS 기본 차단
- 해외/비정상 IP 접근 제한 후보

### 7순위. MFA / SSO / 재인증

필수 정책:

- 관리자 MFA 필수
- 일반 직원 MFA 선택 또는 조건부
- 개인정보 접근 시 재인증
- 중요 결재 시 2차 인증
- 세션 시간 제한
- 로그인 실패 제한
- 퇴사자 계정 즉시 비활성화

후보:

- Google Workspace SSO
- Microsoft Entra ID
- Okta
- PASS/NICE 본인인증
- OTP 앱 인증

### 8순위. 서버 방화벽 / 네트워크 접근통제

필수 정책:

- 외부 공개는 80/443만 허용
- SSH 22 직접 공개 금지 또는 관리자 IP 제한
- SSH Key 로그인
- root 로그인 금지
- DB 포트 외부 차단
- DB는 애플리케이션 서버에서만 접근
- 파일 저장소 공개 버킷 금지
- 임시 서명 URL 사용

### 9순위. 서버 EDR / 백신

후보:

- Microsoft Defender for Endpoint
- AhnLab EDR
- SentinelOne
- CrowdStrike
- Sophos
- Trend Micro Cloud One

서버 적용 항목:

- 실시간 악성코드 탐지
- 랜섬웨어 행위 탐지
- 의심 프로세스 알림
- SSH/RDP 접속 감시
- 관리자 명령 실행 로그
- 보안 패치 상태 점검

### 10순위. 로그 수집 / SIEM / 관제

초기 후보:

- Grafana Loki
- ELK
- Wazuh
- Datadog
- Cloudflare logs 후보

초기에는 대형 SIEM보다 감사로그/접속기록/서버로그를 먼저 안정적으로 남긴다.

## 7. 서버 오픈 전 필수 체크리스트

서버를 외부에 열기 전 반드시 확인한다.

1. HTTPS 적용
2. WAF 적용 또는 Cloudflare 보호 설정
3. SSH 외부 직접접속 차단 또는 IP 제한
4. DB 외부 공개 차단
5. 관리자 MFA
6. 서버 방화벽
7. 백업 자동화
8. 로그 수집
9. 파일 업로드 제한
10. 관리자/일반 계정 분리

## 8. 실제 직원 데이터 투입 전 필수 체크리스트

실제 인사·급여·계약·증빙 데이터를 넣기 전 확인한다.

1. 개인정보 접속기록 저장
2. 개인정보 다운로드 제한
3. 중요정보 암호화
4. 퇴사자 계정 비활성화 절차
5. 관리자 행위 로그
6. 파일 다운로드 로그
7. 복구 테스트
8. 운영 DB와 개발 DB 분리
9. 민감 파일 접근권한 검증
10. 감사로그 조회 권한 분리

## 9. 외부 사용자 접속 전 필수 체크리스트

협력사, 고객사, 사업장 관리자, 지점 담당자, 외부 파트너가 접속하기 전 확인한다.

1. 본인인증
2. API Rate Limit
3. 외부 사용자 권한 분리
4. 사업장별 접근권한
5. 이상 로그인 탐지
6. 외부 사용자 감사로그
7. 개인정보 접근 재인증
8. 외부문의/연락 이력 보관
9. 계약/정산/문의 업무 연결
10. 권한 만료/회수 절차

## 10. 개발에 먼저 반영할 순서

제품이나 보안 프로그램 구매보다 먼저 코드와 DB 설계에 반영해야 하는 것은 아래 5개다.

1. 권한 테이블
2. 감사로그
3. 개인정보 접속기록
4. 파일 권한/다운로드 로그
5. 백업/복구 구조

이 5개는 나중에 덧붙이기 어렵다. 초기 설계가 잘못되면 그룹웨어 전체를 다시 뜯어고쳐야 한다.

## 11. 당장 다음 구현 후보

다음 개발 순서는 아래 중 하나로 잡는 것이 좋다.

### 후보 A. 권한·감사로그·개인정보 접속기록 기반 보강

장점:

- 보안 기반을 가장 먼저 다진다.
- 외부 API를 붙이기 전 내부 데이터 보호 구조가 잡힌다.

구성:

- 개인정보 접근 이벤트 표준화
- 파일 다운로드 감사로그 표준화
- 권한 변경 감사로그 표준화
- 관리자 행위 사유 입력 기준

### 후보 B. R2/S3 파일 저장·권한·다운로드 로그 고도화

장점:

- 전자계약, OCR, 증빙, 문서함의 기반이 된다.
- 파일 보안과 외부 API 연동의 공통 토대다.

구성:

- 파일 메타데이터 표준화
- 다운로드 권한 검사
- 임시 다운로드 URL
- 파일 해시/감사로그
- 업로드 제한 정책

### 후보 C. 전자계약 API 연동 설계

장점:

- 위아히어 업무에 바로 체감되는 기능이다.
- 승인 → 계약 발송 → 서명 완료 → 문서 보관 흐름이 그룹웨어 핵심 가치와 맞다.

구성:

- 계약 템플릿 모델
- 내부 승인 상태 모델
- 외부 전자계약 발송 후보
- Webhook 수신 모델
- PDF 보관 모델
- 계약 만료 알림 후보

### 후보 D. 외부 문의/연락 수신함 설계

장점:

- 카카오/SMS/이메일 문의를 그룹웨어 업무로 끌어오는 구조를 먼저 잡는다.
- 고객사·거래처 연락 누락을 줄일 수 있다.

구성:

- 외부문의함
- 발신자/거래처 매칭
- 문의 유형 분류
- 담당 부서/담당자 배정
- 처리 상태
- 회신 이력
- 계약/정산/시설/운영 업무 연결

## 12. 추천 다음 스텝

추천은 `후보 A → 후보 B → 후보 C → 후보 D` 순서다.

이유:

1. 권한·감사로그·개인정보 접속기록이 먼저 있어야 외부 API를 안전하게 붙일 수 있다.
2. 파일 저장/권한 구조가 잡혀야 전자계약 PDF, 증빙, OCR 파일을 안전하게 보관할 수 있다.
3. 전자계약 API는 위아히어 업무 체감도가 높다.
4. 카카오/SMS/이메일 외부문의 연동은 고객사·거래처 업무 흐름과 연결하되, 먼저 수신함/업무화 모델을 잡아야 한다.

## 13. 별도 승인 게이트

아래는 문서화와 설계는 가능하지만 실행은 별도 승인으로 둔다.

- 유료 API 계약
- 실제 API Key 발급/입력
- production DB 실데이터 연동
- 외부 전자계약 발송
- 실제 문자/카카오 메시지 발송
- 실제 본인인증 요청
- 실제 홈택스/세금계산서 계정 연동
- DNS/custom domain 변경
- 보안 제품 유료 결제
- production backup/restore 실행
- destructive migration
